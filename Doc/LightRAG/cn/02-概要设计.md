## 一、系统架构

### 1.1 设计模式

LightRAG 采用以下核心设计模式：

| 设计模式 | 应用位置 | 说明 |
|----------|----------|------|
| **门面模式（Facade）** | `LightRAG` 主类 | 对外暴露统一的 `insert` / `query` 入口，屏蔽内部流水线复杂度 |
| **工厂模式（Factory）** | `kg/factory.py`、`llm/` | 按配置字符串动态加载存储后端、LLM 提供商 |
| **抽象基类（ABC）** | `base.py` | 定义 `BaseKVStorage` / `BaseVectorStorage` / `BaseGraphStorage` / `DocStatusStorage` 四大存储契约 |
| **注册表（Registry）** | `kg/__init__.py` STORAGES、parser/registry | 存储实现、解析器插件注册与发现 |
| **策略模式（Strategy）** | `chunker/`、查询 `mode` | 四种分块策略、六种查询模式可切换 |
| **异步管道（Async Pipeline）** | `pipeline.py`、`operate.py` | 全异步摄入与检索流水线 |
| **角色分离（Role-based）** | `llm_roles.py` | EXTRACT/QUERY/KEYWORDS/VLM 四角色独立模型配置 |
| **依赖注入** | FastAPI 路由 `Depends(combined_auth)` | 鉴权与配置注入 |

### 1.3 双层检索范式（核心算法思想）
- **低层（Low-level）检索**：以查询抽取的关键词定位具体**实体与关系**，回答细节性问题
- **高层（High-level）检索**：定位**主题与汇总性关键词**，回答需要全局视角的问题
- `hybrid` 模式同时执行两层；`mix` 模式在图检索之上叠加朴素向量召回

## 四、核心流程概要

### 4.1 摄入流程
1. 文档经解析器（Parser）转为统一文本结构
2. 分块器（Chunker）按选定策略切分为 chunk
3. `pipeline.py` 并发调度 `extract_entities` 调用 LLM 抽取实体/关系
4. 向量存储写入 chunk/实体/关系的 embedding
5. `merge_nodes_and_edges` 合并到图存储，更新 KV 与文档状态

### 4.2 查询流程
1. `extract_keywords_from_query` 抽取高/低层关键词
2. 按 `mode` 选择检索路径（local/global/hybrid/mix/naive）
3. 从向量库召回相关实体/关系/chunk
4. 图存储获取子图上下文（`get_knowledge_graph`、`get_node_edges`）
5. 可选 Rerank 重排
6. 拼接上下文调用 LLM 生成答案（支持 SSE 流式）

## 六、技术选型总结
- **语言/运行时**：Python ≥ 3.10，全异步（asyncio）
- **Web**：FastAPI + Uvicorn/Gunicorn
- **前端**：React + Vite（Web UI）
- **包管理**：uv（推荐）/ pip；构建用 setuptools
- **测试**：pytest + pytest-asyncio
- **部署**：Docker / Podman / Kubernetes（含 k8s-deploy 清单）
