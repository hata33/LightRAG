## 一、设计说明

LightRAG **不绑定单一数据库**，而是定义**四大存储抽象**，由不同后端实现。物理形态因后端而异（JSON 文件 / 关系表 / 图数据库 / 向量库 / 文档库），但逻辑数据模型一致。本文档描述统一的**逻辑数据模型**，并给出 PostgreSQL 一体化方案的关系表设计作为参考实现。

### 1.1 四大存储抽象

| 存储类型 | 抽象基类 | 默认实现 | 用途 |
|----------|----------|----------|------|
| KV 存储 | `BaseKVStorage` | `JsonKVStorage` | chunk 文本、实体描述、关系描述的键值存储 |
| 向量存储 | `BaseVectorStorage` | `NanoVectorDBStorage` | chunk/实体/关系的语义向量召回 |
| 图存储 | `BaseGraphStorage` | `NetworkXStorage` | 知识图谱节点与边 |
| 文档状态 | `DocStatusStorage` | `JsonDocStatusStorage` | 文档处理状态追踪 |

## 二、核心逻辑数据模型

### 2.1 文档与 chunk

#### 文档状态记录（doc_status）

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | string | 文档 ID（track_id 关联） |
| content_summary | string | 内容摘要/文件名 |
| content_length | int | 内容长度 |
| status | enum | `PENDING` / `PROCESSING` / `PROCESSED` / `FAILED` |
| chunks_count | int | chunk 数量 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |
| file_path | string | 来源文件路径 |
| metadata | json | 扩展元数据 |

#### chunk 记录（key_string_value_chunk_storage）

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | string | chunk ID（如 `chunk_<sha>`） |
| tokens | int | token 数 |
| content | text | chunk 文本 |
| full_doc_id | string | 所属文档 ID |
| order | int | 在文档中的顺序 |

#### chunk 向量（chunks_vdb）

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | string | chunk ID |
| content | text | 用于召回的文本 |
| vector | vector(dim) | 嵌入向量 |

### 2.2 知识图谱

#### 实体节点（entities）

| 字段名 | 类型 | 说明 |
|--------|------|------|
| entity_id | string | 实体名（归一化后） |
| entity_type | string | 类型（如 organization/person/...） |
| description | text | 实体描述（多源合并） |
| source_id | string | 来源 chunk id 列表 |
| weight | float | 权重（出现/置信度） |
| created_at | datetime | 创建时间 |

#### 实体向量（entities_vdb）

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | string | 实体名 |
| content | text | 实体名+描述 |
| vector | vector(dim) | 嵌入向量 |

#### 关系边（relationships）

| 字段名 | 类型 | 说明 |
|--------|------|------|
| src_id | string | 起点实体 |
| tgt_id | string | 终点实体 |
| description | text | 关系描述 |
| keywords | string | 关系关键词 |
| weight | float | 权重 |
| source_id | string | 来源 chunk id |
| created_at | datetime | 创建时间 |

#### 关系向量（relationships_vdb）

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | string | `src_id##tgt_id` |
| content | text | 关系描述 |
| vector | vector(dim) | 嵌入向量 |

### 2.3 辅助数据
- **LLM 查询缓存**：`(prompt_hash) → (response)`，避免重复 LLM 调用，可独立清理
- **chunk→entity 池**：`chunk_entity_pool`，记录每 chunk 已抽取实体

## 三、表间关系（实体-关系图）

```
doc_status (1) ──< chunk (N) ──< chunk_vector
                    │
                    │ source_id
                    ▼
              entity (N) >── relationship ──< entity (N)
                    │                              │
                    ▼                              ▼
              entity_vector               relationship_vector
```

- 一个文档产生多个 chunk
- 一个 chunk 抽取产生多个实体与关系
- 实体间通过关系边连接（有向图）
- 实体、关系、chunk 各自维护独立的向量索引

## 四、PostgreSQL 一体化方案（参考物理表设计）

PostgreSQL（+ pgvector + AGE 图扩展）可作为四合一统一后端，物理表设计：

```sql
-- chunk KV 存储
CREATE TABLE LIGHTRAG_DOC_CHUNKS (
    id TEXT PRIMARY KEY,
    content TEXT NOT NULL,
    tokens INTEGER,
    full_doc_id TEXT,
    chunk_order INTEGER
);

-- chunk 向量
CREATE TABLE LIGHTRAG_DOC_FULL (
    id TEXT PRIMARY KEY,
    content TEXT,
    vector VECTOR(<dim>)
);

-- 文档状态
CREATE TABLE LIGHTRAG_DOC_STATUS (
    id TEXT PRIMARY KEY,
    content_summary TEXT,
    content_length BIGINT,
    status TEXT,
    chunks_count INT,
    created_at TIMESTAMPTZ,
    updated_at TIMESTAMPTZ,
    file_path TEXT,
    metadata JSONB
);

-- 实体向量
CREATE TABLE LIGHTRAG_VDB_ENTITY (
    id TEXT PRIMARY KEY,
    content TEXT,
    vector VECTOR(<dim>)
);

-- 关系向量
CREATE TABLE LIGHTRAG_VDB_RELATION (
    id TEXT PRIMARY KEY,
    content TEXT,
    vector VECTOR(<dim>)
);
```

> 知识图谱在 PG 方案中由 **Apache AGE** 图扩展承载（节点/边为 Cypher 查询），或退化为关系表 + JSON 邻接。MongoDB / OpenSearch 方案对应为各自的文档结构。

## 五、索引建议

| 表/集合 | 索引字段 | 索引类型 | 说明 |
|---------|----------|----------|------|
| chunk | full_doc_id | B-Tree | 按文档反查 chunk |
| doc_status | status | B-Tree | 按状态筛选（失败重处理） |
| chunk_vector / entity_vector / relation_vector | vector | HNSW / IVFFlat | 语义近似检索（pgvector） |
| entities | entity_id | 唯一索引 / 主键 | 实体去重合并 |
| relationships | (src_id, tgt_id) | 复合索引 | 边存在性检查与遍历 |
| relationships | keywords | GIN / 全文索引 | 关键词检索 |
| LLM 缓存 | prompt_hash | 唯一索引 | 缓存命中查找 |

## 六、后端选型矩阵（数据存储维度）

| 抽象 | 轻量/默认 | 生产关系型 | 向量专用 | 图专用 | 文档型 | 全文/统一 |
|------|-----------|-----------|----------|--------|--------|-----------|
| KV | JSON | PostgreSQL | - | - | MongoDB | Redis / OpenSearch |
| 向量 | NanoVectorDB | PG+pgvector | Milvus / Faiss / Qdrant | - | Mongo | OpenSearch |
| 图 | NetworkX | PG+AGE | - | Neo4j / Memgraph | Mongo | OpenSearch |
| 状态 | JSON | PostgreSQL | - | - | Mongo | Redis / OpenSearch |

- **一键统一后端**：PostgreSQL（+pgvector+AGE）、MongoDB、OpenSearch 均可单独承载全部四类存储
- **环境变量要求**：见 `kg/__init__.py` 的 `STORAGE_ENV_REQUIREMENTS`（如 `MONGO_URI`、`POSTGRES_*`、`NEO4J_*`、`MILVUS_URI`、`QDRANT_URL`、`OPENSEARCH_HOSTS`）
