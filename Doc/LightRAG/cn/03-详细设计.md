## 一、LightRAG 主类详细设计

`LightRAG`（`lightrag/lightrag.py`，门面类）是系统的统一入口，持有四类存储、LLM 函数、配置等依赖。

### 1.1 摄入方法 `ainsert` / `insert`

**方法签名**：
```python
async def ainsert(self, input: str | list[str], ...) -> None
def insert(self, input, ...) -> None   # 同步包装，内部跑事件循环
```

**算法流程**：
1. **预处理**：将输入统一为字符串列表，对每个 input 调用解析与分块
2. **分块**：依据 `chunking_strategy`（token_size/recursive/semantic/paragraph）调用对应 chunker 生成 chunks
3. **入队流水线**：将 chunks 连同文档元数据写入文档状态存储（状态置为 `PROCESSING`），调用 `pipeline.py` 编排
4. **实体抽取**（并发）：`operate.extract_entities` 对每个 chunk 调用 LLM（EXTRACT 角色），解析得到实体、关系记录
5. **向量入库**：将 chunk 文本、实体名、关系描述分别 embedding 后 upsert 到对应向量库
6. **图合并**：`operate.merge_nodes_and_edges` 将新抽取的节点/边去重合并到图存储，更新 type/source_id/weight 等字段
7. **状态收尾**：文档状态置为 `PROCESSED`，触发回调（`index_done_callback`）刷盘

**关键分支**：
- 已存在相同文档（文档状态命中）→ 跳过或按策略增量处理
- LLM 抽取返回非法 JSON → `json_repair` 修复，仍失败则重试（tenacity）
- 并发上限控制：通过信号量/并发参数限制 `extract_entities` 并发度

### 1.2 查询方法 `aquery` / `query`

**方法签名**：
```python
async def aquery(self, query: str, param: QueryParam) -> str | AsyncIterator
```

**算法流程（按 QueryParam.mode 分支）**：
1. **关键词抽取**：`operate.extract_keywords_from_query` 调用 LLM（KEYWORDS 角色）得到高层与低层关键词
2. **检索分支**：
   - `local`：仅低层关键词 → 召回相关实体 → 图存储取邻居关系/chunk
   - `global`：仅高层关键词 → 召回相关关系/主题 chunk
   - `hybrid`：低层 + 高层合并召回
   - `mix`：图检索（local/global 路径）+ naive 向量召回叠加
   - `naive`：仅对 chunk 向量库做相似度召回
   - `bypass`：跳过检索，直接将 query 交 LLM
3. **上下文构建**：`kg_query` 聚合召回的实体描述、关系描述、chunk 文本为上下文
4. **重排序（可选）**：`rerank` 对候选 chunk 重排取 Top-K
5. **生成**：拼接提示词（QUERY 角色 LLM），支持 SSE 流式 yield
6. **引用**：按配置附加来源 chunk id 与文档 id

### 1.3 图谱管理方法
- `acreate_entity` / `acreate_relation`：校验存在性后写入图存储
- `aedit_entity` / `aedit_relation`：读取 → 合并字段 → upsert
- `amerge_entities`：多实体合并，迁移入边/出边，删除被合并节点
- `adelete_by_doc_id`：按文档删除，遍历受影响实体，重连残留关系，清理向量与 KV
- `adelete_by_entity` / `adelete_by_relation`：单点删除并维护图一致性
- `aget_processing_status`：聚合文档状态计数

### 1.4 自定义摄入
- `ainsert_custom_chunks`：跳过解析/分块，直接对给定 chunk 走抽取与图构建
- `ainsert_custom_kg`：跳过 LLM 抽取，直接注入预构建实体/关系并合并入库

## 二、pipeline.py 摄入编排详细设计

`pipeline.py` 负责把"一批 chunk"编排为完整的知识图谱构建任务，包含：
- **chunk 入库**：写 `key_string_value_chunk_storage` 与 `chunks_vdb`
- **并发抽取调度**：按并发参数分批调用 `extract_entities`
- **去重缓存**：`chunk_entity_pool` 缓存已处理 chunk 的实体池
- **合并提交**：抽取完成后批量调用 `merge_nodes_and_edges`
- **失败隔离**：单 chunk 抽取失败不阻断整批，标记失败供 `/reprocess_failed` 重试

**关键分支**：
- 大规模数据 → 分批 + 并发以避免内存/限流瓶颈
- 多 Worker 部署 → 通过 `shared_storage` 协调，索引完成回调刷盘后才对其他 Worker 可见

## 三、operate.py 检索与抽取算法

### 3.1 `extract_entities`
```
输入: chunks, llm_func, prompt_template, max_gleaning
1. for chunk in chunks (并发):
2.     resp = llm_func(entity_prompt + chunk)
3.     parsed = json_repair.loads(resp)         # 容错 LLM 输出
4.     entities, relations = normalize(parsed)
5.     for _ in range(max_gleaning):             # 多轮 gleaning 增补
6.         resp = llm_func(gleaning_prompt)
7.         merge into entities/relations
8. return entities, relations
```
时间复杂度：O(N × G) 次额外 LLM 调用（N=chunk 数，G=gleaning 次数）。

### 3.2 `merge_nodes_and_edges`
- 按 entity name 归并；type/source_id 取并集；weight/description 累加或按规则合并
- 边按 (src, tgt) 归并；weight 累加、description 去重拼接
- 批量 upsert 到图存储（`upsert_nodes_batch`、`upsert_edges_batch`）

### 3.3 `kg_query`
```
输入: keywords(low/high), mode, vdb, graph, top_k
1. 候选实体 = vdb.query(low_keywords)  或 high_keywords
2. 子图上下文 = graph.get_node_edges / get_knowledge_graph(候选实体)
3. 候选 chunk = vdb.query(对应向量)
4. (mix 模式) 叠加 naive 召回
5. context = 聚合(实体描述, 关系描述, chunk 文本)
6. return context
```

### 3.4 `naive_query`
直接对 `chunks_vdb` 做相似度召回 Top-K，拼接为上下文。

### 3.5 关键词提取 `extract_keywords_from_query`
- 调用 LLM（KEYWORDS 角色）输出 JSON：`{low_level_keywords:[], high_level_keywords:[]}`
- 失败容错：降级为简单分词

## 四、关键逻辑分支与边界

### 4.1 多模态处理
- 当文档含图片/表格/公式 → 走 `multimodal_context.py` + `prompt_multimodal.py`
- VLM 角色 LLM 负责图像理解；表格通过 `_html_table` / parser 提取

### 4.2 流式输出分支
- `/query/stream` → 返回 `AsyncIterator[str]`，逐 token yield
- 普通 `/query` → 聚合为完整字符串

### 4.3 删除一致性
- 删除文档时必须保证：图存储节点、边、向量库条目、KV chunk、文档状态记录四者一致
- 通过先标记后清理、批量删除、重连边来维护

### 4.4 历史对话
- `QueryParam.history_turns > 0` → 将历史问答拼入上下文，支持多轮

## 五、并发与一致性设计
- 全异步：所有存储方法为 `async`
- 并发限流：信号量控制 LLM 并发、批量接口降低往返
- 写入原子：`file_atomic` 保证 JSON 存储并发写安全
- 多 Worker：`shared_storage` + 完成回调保证索引可见性
