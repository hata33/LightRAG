## 二、核心抽象：四类存储契约

整个系统建立在**四类存储抽象**之上。这是 LightRAG 最核心的设计——所有后端都实现同一套契约，业务代码完全不感知底层是 JSON 文件还是 PostgreSQL。

```mermaid
graph TB
    subgraph CONTRACT["base.py · 存储抽象基类（契约）"]
        direction TB

        KV["BaseKVStorage<br/>━━━━━━━━━━━━━<br/>get_by_id / get_by_ids<br/>upsert / delete<br/>━━━━━━━━━━━━━<br/>用途: 文档/块/实体/关系/缓存"]
        VEC["BaseVectorStorage<br/>━━━━━━━━━━━━━<br/>query / upsert<br/>delete_entity / delete_relation<br/>━━━━━━━━━━━━━<br/>用途: 向量召回"]
        GRAPH["BaseGraphStorage<br/>━━━━━━━━━━━━━<br/>upsert_node / upsert_edge<br/>get_node / has_edge<br/>get_node_edges / get_knowledge_graph<br/>━━━━━━━━━━━━━<br/>用途: 知识图谱"]
        STATUS["DocStatusStorage<br/>继承 BaseKVStorage<br/>━━━━━━━━━━━━━<br/>get_docs_by_status<br/>━━━━━━━━━━━━━<br/>用途: 文档状态机"]
```

### 契约的核心方法（业务代码只调这些）

| 抽象类 | 业务调用的方法 | 语义 |
|---|---|---|
| `BaseKVStorage` | `get_by_id(id)` / `upsert(data)` / `get_by_ids(ids)` | 按 ID 存取结构化数据 |
| `BaseVectorStorage` | `query(query_vector, top_k)` / `upsert(data)` | 向量相似度召回 |
| `BaseGraphStorage` | `upsert_node()` / `upsert_edge()` / `get_node_edges()` / `get_knowledge_graph()` | 图节点/边读写与遍历 |
| `DocStatusStorage` | `get_docs_by_status(status)` | 按状态查文档（继承 KV） |

## 四、三个可插拔轴

LightRAG 的架构灵活性来自三个独立的可插拔轴，每个轴都能单独替换：

```mermaid
graph LR
    HUB["operate.py + pipeline.py<br/>业务核心（不变）"]

    subgraph AXIS1["轴 1 · 存储后端"]
        A1_NORM["默认: JSON+NanoVectorDB+NetworkX<br/>零依赖单机"]
        A1_PROD["生产: PostgreSQL 全合一<br/>pgvector + AGE"]
        A1_BIG["大规模: Neo4j+Milvus<br/>专用组合"]
    end

    subgraph AXIS2["轴 2 · LLM/Embedding"]
        A2_OAI["OpenAI / Claude / Gemini"]
        A2_LOCAL["Ollama / vLLM 本地"]
        A2_CN["智谱 GLM / Bedrock"]
    end

    subgraph AXIS3["轴 3 · 解析引擎"]
        A3_LOCAL["legacy / native 本地"]
        A3_MINERU["mineru OCR+版面"]
        A3_DOCLING["docling IBM"]
    end

    AXIS1 -->|"4 抽象基类<br/>工厂动态加载"| HUB
    AXIS2 -->|"4 角色路由<br/>适配器模式"| HUB
    AXIS3 -->|"registry 注册<br/>插件化"| HUB

    style HUB fill:#fffde7,stroke:#f57f17
    style AXIS1 fill:#e3f2fd,stroke:#1565c0
    style AXIS2 fill:#f3e5f5,stroke:#6a1b9a
    style AXIS3 fill:#e8f5e9,stroke:#2e7d32
```

### 三个轴的替换方式

| 轴 | 替换方式 | 契约 |
|---|---|---|
| **存储** | `kv_storage=` / `vector_storage=` 等参数，或 `LIGHTRAG_*_STORAGE` 环境变量 | `base.py` 4 个抽象类 |
| **模型** | `llm_model_func=` / `embedding_func=` 参数，或 `*_BINDING` 环境变量 | `llm_roles.py` 4 角色 + `EmbeddingFunc` 装饰器 |
| **解析器** | `LIGHTRAG_PARSER` 路由规则 + `register_parser()` API | `BaseParser.parse(ctx)` + `ParserSpec` |

## 五、数据架构：存储如何组织知识

LightRAG 把一份文档拆解成**四类向量 + 一个图 + 五类 KV**，形成立体知识结构：

```mermaid
graph TB
    DOC["📄 一份文档"]

    DOC --> PARSED["解析: full_docs<br/>原文 + Markdown"]
    DOC --> STATUS["状态: doc_status<br/>PENDING→PROCESSED"]

    PARSED --> CHUNKED["分块: text_chunks + chunks_vdb<br/>文本块 + 块向量"]

    CHUNKED --> EXTRACTED["抽取: 实体 + 关系"]

    EXTRACTED --> ENT_VEC["entities_vdb<br/>实体向量<br/>━━━━━━━<br/>用于: 召回相关实体"]
    EXTRACTED --> REL_VEC["relationships_vdb<br/>关系向量<br/>━━━━━━━<br/>用于: 召回相关关系"]
    EXTRACTED --> ENT_DETAIL["full_entities<br/>实体详情<br/>━━━━━━━<br/>name/desc/source_id"]
    EXTRACTED --> REL_DETAIL["full_relations<br/>关系详情<br/>━━━━━━━<br/>src/tgt/desc/weight"]
    EXTRACTED --> KG["chunk_entity_relation_graph<br/>知识图谱<br/>━━━━━━━<br/>节点(实体)+边(关系)"]

    CACHE["llm_response_cache<br/>LLM 调用缓存<br/>━━━━━━━<br/>抽取/查询结果复用"]

    style DOC fill:#fffde7,stroke:#f57f17
    style KG fill:#e8f5e9,stroke:#2e7d32
    style ENT_VEC fill:#e3f2fd,stroke:#1565c0
    style REL_VEC fill:#e3f2fd,stroke:#1565c0
    style CHUNKED fill:#fff3e0,stroke:#e65100
```

### 为什么是「图 + 三向量」而非单一向量库

```mermaid
graph LR
    subgraph 传统["传统向量 RAG"]
        T1["文档→chunk→单向量库<br/>只能算 chunk 相似度"]
        T1 --> T2["❌ 多跳推理弱<br/>❌ 实体关系丢失"]
    end

    subgraph LIGHTRAG["LightRAG 立体知识"]
        L1["chunk 向量 → 找相关段落"]
        L2["entity 向量 → 找相关实体"]
        L3["relation 向量 → 找相关关系"]
        L4["graph 遍历 → 多跳推理"]
        L1 & L2 & L3 & L4 --> L5["✅ 融合检索<br/>local + global + hybrid + mix"]
    end

    style 传统 fill:#ffcdd2
    style LIGHTRAG fill:#c8e6c9
```

## 十、架构决策记录（ADR 速览）

| # | 决策 | 理由 |
|---|---|---|
| 1 | 四类存储抽象 + 工厂动态加载 | 业务代码与后端完全解耦，按规模平滑演进 |
| 2 | 图 + 三向量立体知识结构 | 克服单向量库的多跳推理短板 |
| 3 | 双层检索范式（local + global） | 兼顾细节事实与全局综述 |
| 4 | 4 角色 LLM 独立配置 | 抽取用强模型、关键词用快模型，质量成本兼得 |
| 5 | 全异步 + 三级队列 + keyed lock | 大规模并发不锁死 |
| 6 | dataclass 门面 + Mixin 组合 | 公共 API 稳定，内部实现可拆分演进 |
| 7 | 解析器 registry 插件化 | 第三方引擎可 `register_parser()` 接入 |
| 8 | 惰性导入（importlib） | 不用的后端/引擎不占启动时间和内存 |
