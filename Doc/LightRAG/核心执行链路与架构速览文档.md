# 核心执行链路与架构速览文档

**项目**：LightRAG · **版本**：1.5.5（文档版本 1.26.705.8600）· **日期**：2026-07-05 · **作者**：15531

---

## 1. 一句话定位

LightRAG 是一个把**知识图谱**与 LLM 深度融合的轻量级 RAG 框架，通过**双层检索（低层实体 / 高层主题）** 在私有知识库上实现比传统向量检索更准确的多跳推理与全局综述问答，并提供可插拔的存储/模型后端与开箱即用的 API 服务。

---

## 2. 骨架调用图

### 2.1 摄入链路（从 `LightRAG.ainsert` / 服务端 `apipeline_enqueue_documents` 入口）

```
rag.ainsert(text)                              # 门面入口（SDK，固定 token 分块）
└─→ rag.apipeline_enqueue_documents(...)        # 或服务端 POST /documents/text → 此处（支持 F/R/V/P 分块）
    ├─→ parser.parse(file)                      # 文档解析（native / mineru / docling / legacy）
    ├─→ chunker.chunking_by_*(text)             # 分块策略 F/R/V/P（strategy 模式）
    ├─→ doc_status.upsert(PENDING→PROCESSING)   # 文档状态机
    └─→ pipeline.process_single_document(...)   # 单文档编排
        ├─→ operate.extract_entities(chunks)    # LLM 抽取实体/关系（并发 + gleaning）
        │       └─→ role_llm["extract"].complete()  # EXTRACT 角色 LLM
        ├─→ vdb.upsert(embeddings)              # chunk/实体/关系向量入库
        └─→ operate.merge_nodes_and_edges()     # 去重合并到知识图谱
                └─→ graph.upsert_node / upsert_edge()
        └─→ doc_status.upsert(PROCESSED) + index_done_callback()  # 刷盘
```

### 2.2 查询链路（从 `LightRAG.aquery` / `POST /query` 入口）

```
rag.aquery(query, QueryParam(mode))            # 门面入口
└─→ operate.extract_keywords_from_query(query)  # 关键词抽取（KEYWORDS 角色 LLM）
│       └─→ {low_level_keywords, high_level_keywords}
├─→ [mode 分支]
│   ├ local   → operate.kg_query(LL 关键词)    # 实体级检索
│   ├ global  → operate.kg_query(HL 关键词)    # 主题级检索
│   ├ hybrid  → operate.kg_query(LL + HL)      # 双层
│   ├ mix     → kg_query + naive_query 叠加    # 图 + 向量（默认）
│   ├ naive   → operate.naive_query()          # 纯向量召回
│   └ bypass  → 跳过检索
│       │
│       └─→ kg_query 内部:
│           ├─→ entities_vdb / relationships_vdb.query()   # 向量召回实体/关系
│           ├─→ graph.get_node_edges / get_knowledge_graph()  # 图遍历取子图
│           ├─→ chunks_vdb.query()                         # 召回相关 chunk
│           └─→ rerank.rerank(candidates)                  # 可选重排
└─→ role_llm["query"].complete(context + query)  # 生成答案（QUERY 角色，支持 SSE 流式）
```

## 4. 核心设计决策

| # | 设计决策 | 带来的好处 |
|---|----------|-----------|
| 1 | **四类存储抽象（KV/Vector/Graph/DocStatus）+ 工厂注册表动态加载** | 存储后端完全可插拔：同一份业务代码可在 JSON 文件（单机）↔ PostgreSQL/MongoDB/OpenSearch（统一）↔ Neo4j+Milvus（专用）之间无缝切换，按规模演进而无需改业务代码 |
| 2 | **双层检索范式（低层实体 + 高层主题）融入知识图谱** | 既擅长细节事实（local）又擅长全局综述（global），hybrid/mix 模式覆盖传统向量 RAG 的多跳推理短板 |
| 3 | **角色化 LLM（extract/keyword/query/vlm）独立配置** | 不同环节可用不同模型/端点：抽取用强模型保质量、关键词用快模型省成本、VLM 处理多模态，实现质量与成本的精细权衡 |
| 4 | **全异步（asyncio）+ 批量接口 + 信号量并发控制** | 大规模语料高效处理，并发可控不击穿下游 LLM/DB，配合 `shared_storage` 支持多 Worker 水平扩展 |
| 5 | **配置三路注入（env > 构造参数 > config.ini）+ 可观测 addon_params** | 12-factor 友好，适合容器化部署；角色/分块/并发等参数可热感知，便于运维调优 |

## 5. 最简示例运行路径

```python
import asyncio
from lightrag import LightRAG, QueryParam

async def main():
    rag = LightRAG(working_dir="./rag_storage")      # ① 构造（门面）
    await rag.initialize_storages()                  # ② 初始化四类存储
    await rag.ainsert("LightRAG 用知识图谱做双层检索。")  # ③ 摄入 → 见骨架 2.1
    ans = await rag.aquery(                          # ④ 查询 → 见骨架 2.2
        "LightRAG 的检索范式是什么？",
        param=QueryParam(mode="hybrid"),
    )
    print(ans)

asyncio.run(main())
```

**执行流程对应骨架调用图节点**：
- `①` → `LightRAG` 门面（第 3 节门面模块）
- `②` → 触发 `kg/factory.get_storage_class()` 按配置实例化四类后端（第 3 节存储后端）
- `③` → 骨架 **2.1** 全链路：`ainsert → apipeline_enqueue_documents → parser → chunker → extract_entities(EXTRACT 角色) → vdb.upsert → merge_nodes_and_edges → graph.upsert`
- `④` → 骨架 **2.2** 全链路：`aquery → extract_keywords_from_query(KEYWORDS) → kg_query(hybrid) → vdb/graph 检索 → rerank → role_llm["query"].complete`

> 服务端等价路径：`POST /documents/text` → 骨架 2.1；`POST /query` → 骨架 2.2。
