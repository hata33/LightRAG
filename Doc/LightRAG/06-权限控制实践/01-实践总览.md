# 实践总览 —— 企业知识库权限控制

**项目**: LightRAG · **日期**: 2026-07-12 · **作者**: 15531

> 工作职责:企业知识库 RAG 检索与问答系统,对标 RAGFlow,实现从文档解析、切片、向量化到语义检索与生成问答的完整链路,支撑多部门 1000+ 份技术文档的即时问答。

---

## 一、背景

LightRAG 是一个研究级单租户 RAG 框架。要支撑企业级场景(多部门、多角色、文档级权限),必须解决一个核心问题:

**不同角色看到的和检索到的内容范围必须受控。**

### 1.1 企业知识库的权限需求

| 场景 | 要求 |
|---|---|
| 财务部 vs 工程部 | 部门间数据完全隔离(财务文档不能泄露给工程部) |
| 同部门内 财务月报 vs 薪资表 | 同一部门内不同文档不同权限(普通员工能看月报,不能看薪资) |
| LLM 生成的回答 | 不能在回答文本中泄露越权内容(不能出现薪资数字) |

### 1.2 为什么 Graph RAG 的权限特别难

```
传统 RAG(朴素向量检索):
  query → 向量搜索 chunk → 返回
  权限只需:在向量搜索时按 doc_id 过滤即可 ✅ 简单

Graph RAG:
  query → 向量搜索 entity/relation → 沿知识图谱多跳扩展
         → 用 entity/relation 反查关联 chunk → 组装上下文 → LLM 生成
  权限难点:
  1. entity/relation 是跨文档共享的 ——「苹果公司」可能来自 100 个文档
  2. 一个 relation 的 file_path 可能来自多个不同权限的文档
  3. 知识图谱的多跳扩展可能「绕过」权限边界
  4. 向量检索是「匿名的」—— 返回 top-k 不带权限标签
```

**核心矛盾**:实体/关系天然是跨文档聚合的,但权限是按文档划分的。

## 二、三层防御架构

经过渐进式探索,最终采用了三层防御:

```
┌─────────────────────────────────────────────────────────────┐
│  第 1 层:workspace 级隔离(物理隔离)            练习 1     │
│  alice → tenant_finance     (独立 LightRAG 实例)            │
│  bob   → tenant_engineering (独立 LightRAG 实例)            │
│  → 不同租户数据完全隔离,不可能串                            │
└─────────────────────────────────────────────────────────────┘
          │
          ▼ 同一 workspace 内,再做文档级控制
┌─────────────────────────────────────────────────────────────┐
│  第 2 层:文档级 ACL pre-filter(检索层过滤)     练习 3     │
│  代理注入 ACL 白名单 → 后端检索时就排除越权文档             │
│  → LLM 根本看不到越权内容 → 文本天然干净                    │
└─────────────────────────────────────────────────────────────┘
          │
          ▼ 作为双保险
┌─────────────────────────────────────────────────────────────┐
│  第 3 层:文档级 ACL post-fetch(输出层过滤)     练习 2     │
│  后端返回结果后,代理再过滤 references/chunks               │
│  → 防御纵深,即使 pre-filter 有遗漏也能兜底                 │
└─────────────────────────────────────────────────────────────┘
```

### 为什么需要三层?

每一层解决不同层面的威胁:

| 层 | 解决的问题 | 单独的局限 |
|---|---|---|
| workspace 隔离 | 跨部门数据泄漏 | 同部门内无法做文档级权限 |
| pre-filter | LLM 文本泄露越权内容 | 需要改 LightRAG 核心,有 upstream 冲突 |
| post-fetch | pre-filter 的遗漏 | **LLM 文本可能已包含越权内容**(只过滤引用) |

## 三、技术选型决策

### 3.1 为什么选择「反向代理」而非「改核心」

LightRAG fork 自活跃的 upstream(`HKUDS/LightRAG`),持续合入 PR。大改核心的风险:

| 方案 | upstream 冲突 | 安全强度 | 工作量 |
|---|---|---|---|
| 改 operate.py 核心 | ❌ 极高 | ★★★★★ | 15-17天 |
| **反向代理 + 后端多实例** | ✅ 零 | ★★★★☆ | 5-7天 |
| 纯 SDK 包装 | ✅ 零 | ★★☆☆☆ | 3-5天 |

最终选择:**反向代理(零侵入)+ 向量层 pre-filter(feature branch 上改核心,不 merge main)**。

### 3.2 为什么用 ReBAC 而非 RBAC

| 模型 | 代表 | 优势 | 本项目选择 |
|---|---|---|---|
| RBAC | 角色→权限 | 简单 | 不够灵活(无法表达"alice 是 finance_report 的 owner") |
| **ReBAC** | Google Zanzibar / SpiceDB | 关系图、可继承、细粒度 | ✅ 选择 |

ReBAC 的核心是 **relationship 三元组**:
```
document:finance_report #viewer @ user:alice
```
表示「alice 是 finance_report 的 viewer」。权限由关系图计算:
```
permission view = viewer + owner
```

### 3.3 关于 SpiceDB 的决策

原计划用 Docker 运行 SpiceDB 容器,但镜像拉取超时(网络问题)。最终用**纯 Python 实现了 ReBAC 引擎**,接口与 authzed-py 一致(`check_permission` / `write_relationships` / `lookup_resources`)。以后 Docker 可用时,换一行 import 即可切到真 SpiceDB。

## 六、坑位复盘 —— 8 个坑及修复

> 这些坑是在实践中真实遇到的。每个坑都值得记录,因为它们揭示了分布式系统的复杂性。

### 坑位 1:subprocess stdout 管道阻塞

**现象**:后端子进程启动后 hang 住,`_wait_ready` 超时。

**根因**:用 `stdout=subprocess.PIPE` 但没有及时读取,Pipe 缓冲区满后子进程 write 阻塞。

**修复**:改为重定向到日志文件(`stdout=open(log_path, "w")`),不阻塞。

### 坑位 2:pending.discard 类型错误

**现象**:`_wait_ready` 无限循环,后端明明已就绪但循环不退出。

**根因**:`pending` 集合存的是 workspace 名,但 `discard(port)` 删的是端口号 —— 永远删不掉。

**修复**:`pending.discard(workspace)`。

### 坑位 3:旧进程占用端口

**现象**:新代理启动报 `Errno 10048: 端口已被占用`。

**根因**:上一次的代理进程没杀干净,8000 端口被旧进程占着。新代理退出,但旧代理还在跑(没有 ACL 代码)。

**修复**:启动前 `netstat -ano | grep :8000 | taskkill`。

### 坑位 4:aquery_data clone 漏字段

**现象**:加 `acl_allowed_doc_ids` 到 QueryParam 后,aquery_data 路径的 ACL 白名单丢失。

**根因**:`lightrag.py:2249` 手工 clone QueryParam,只复制了 15 个字段,漏了新增的第 16 个。

**修复**:clone 时加 `acl_allowed_doc_ids=param.acl_allowed_doc_ids`。

**教训**:在手工 clone 的 dataclass 上加字段,必须 grep 所有构造点。

### 坑位 5:LLM 缓存跨 ACL 污染

**现象**:场景1(无 ACL)的答案被场景2(有 ACL)复用,薪资泄露。

**根因**:`compute_args_hash` 算 cache key 时没包含 `acl_allowed_doc_ids`。同样的 query 文本,不同 ACL 命中同一缓存。

**修复**:kg_query 和 naive_query 两处 cache key 都加 `str(sorted(acl_allowed_doc_ids))`。

**教训**:任何影响 LLM 输出的参数,都必须进入 cache key。

### 坑位 6:Content-Length 不匹配

**现象**:代理转发时报 `LocalProtocolError: Too much data for declared Content-Length`。

**根因**:代理改写了请求体(注入 ACL 白名单后 body 变长),但原始 Content-Length 还是旧值。

**修复**:转发时删掉 Content-Length header,让 httpx 自己重算。

**教训**:改写 HTTP body 后必须同步更新 Content-Length,或让 HTTP 客户端自动计算。

### 坑位 7:PG 后端 SQL 缺 file_path

**现象**:pre-filter 把所有 entity/relation 过滤掉(22→0),有权限的查询也返回 no-context。

**根因**:PG 的 `postgres_impl.py` SQL 模板中,relationships/entities query 只 SELECT 了 `src_id/tgt_id/entity_name`,**没有 file_path**。向量搜索结果根本没有 file_path 字段,ACL 过滤函数拿到空字符串,全部判定为无权限。

**修复**:SQL 加 `file_path` 字段。

**教训**:不同后端(NanoVectorDB vs PostgreSQL)返回的字段可能不同。NanoVectorDB 返回所有 meta_fields,但 PG 的 SQL 是手写的,只返回固定字段。跨后端验证很重要。

### 坑位 8:file_path 多值格式

**现象**:entity 的 file_path 不是单个文件名,直接用 `==` 比较永远匹配不上。

**根因**:`file_path` 是 `<SEP>` 拼接的多个文件名(graph RAG 的实体是跨文档聚合的):
```
finance_report.txt<SEP>salary_table.txt
```

**修复**:split(GRAPH_FIELD_SEP) 后逐个检查,用 OR 语义(只要有一个文件在白名单就保留)。

## 七、坑位识别的四个习惯

```
习惯 1:字段同步检查
  加了新字段 → grep 所有构造/clone 这个类的地方 → 逐个确认是否需要同步

习惯 2:缓存 key 完整性
  任何影响结果的参数 → 必须进入 cache key → 否则跨上下文污染

习惯 3:数据格式确认
  用一个字段前 → 先搞清楚它是单值还是多值、什么编码、什么分隔符

习惯 4:同类搜索
  修了一个 bug → grep 找所有同类调用点 → 不修完不算修完
```
