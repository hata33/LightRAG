# 全链路打通:代理注入 + 后端 pre-filter + post-fetch 双保险

**目标**:把练习 1-2-3 串成一个端到端系统,通过 HTTP 代理即可获得完整的三层防御。

---

## 一、架构

```
客户端 POST /query
  Authorization: Bearer <JWT>
    │
    ▼
代理 proxy.py (:8000)
  ├─ 1. JWT 验签 → user_id = "alice"
  ├─ 2. workspace 路由 → tenant_finance → port 9621
  ├─ 3. 查权限引擎 → alice 可看 {finance_report, finance_report_2025}
  ├─ 4. 注入请求体: acl_allowed_doc_ids = ["finance_report", "finance_report_2025"]
  ├─ 5. 转发给后端(删 Content-Length,让 httpx 重算)
  │
    ▼
LightRAG 后端 (:9621)
  ├─ 6. QueryRequest.to_query_params() → list 转 set → QueryParam.acl_allowed_doc_ids
  ├─ 7. _get_node_data / _get_edge_data / _get_vector_context
  │     ├─ 向量搜索 top_k × 3(放大召回)
  │     ├─ 按 file_path 过滤(PG SQL 现在返回 file_path 了)
  │     └─ 排除不在白名单的 entity/relation/chunk
  ├─ 8. LLM 只看到有权限的内容 → 生成干净回答
  │     cache key 含 ACL hash → 不同 ACL 不命中同一缓存
  │
    ▼
代理 proxy.py
  ├─ 9. post-fetch 双保险:过滤 references/chunks
  └─ 10. 返回干净结果
```

## 二、新增改动(在全链路阶段)

### 2.1 QueryRequest 加 ACL 字段(`query_routes.py`)

```python
class QueryRequest(BaseModel):
    ...
    acl_allowed_doc_ids: Optional[list[str]] = Field(
        default=None,
        description="Typically injected by a reverse proxy. "
        "Clients should NOT set this directly (security risk).",
    )

    def to_query_params(self, is_stream: bool) -> QueryParam:
        request_data = self.model_dump(exclude_none=True, ...)
        # list[str] → set[str](QueryParam 需要 set 类型)
        if "acl_allowed_doc_ids" in request_data:
            request_data["acl_allowed_doc_ids"] = set(request_data["acl_allowed_doc_ids"])
        return QueryParam(**request_data)
```

### 2.2 代理注入 ACL 白名单(`proxy.py`)

```python
async def _forward_with_acl(request, user_id, path, filter_fn=None):
    # 层1: pre-filter —— 注入 ACL 白名单
    engine = get_engine()
    allowed_doc_ids = engine.get_viewable_documents(user_id)
    if allowed_doc_ids is not None:
        body_json = json.loads(body)
        body_json["acl_allowed_doc_ids"] = list(allowed_doc_ids)
        body = json.dumps(body_json).encode()

    # 层2: post-fetch —— 后端返回后再过滤(双保险)
    resp = await client.request("POST", url, ...)
    if filter_fn and resp.status_code == 200:
        resp_json = filter_fn(user_id, resp.json())
        return Response(content=json.dumps(resp_json).encode(), ...)
```

## 三、踩的坑

### 坑位 6:Content-Length 不匹配

**现象**:`LocalProtocolError: Too much data for declared Content-Length`

**根因**:代理改写了请求体(注入 ACL 后 body 变长),但原始 Content-Length 还是旧值。

**修复**:转发时跳过 Content-Length header:
```python
for key, value in request.headers.items():
    if key.lower() == "content-length":
        continue  # 改写 body 后必须删,让 httpx 重算
    fwd_headers[key] = value
```

### 坑位 7:PG 后端 SQL 缺 file_path

**现象**:pre-filter 把所有 entity/relation 过滤掉(22→0)。

**根因**:PG 的 SQL 模板 relationships/entities query 只 SELECT 了 `src_id/tgt_id/entity_name`,**没有 file_path**。

**修复**:
```sql
-- 改前
SELECT source_id AS src_id, target_id AS tgt_id, ... FROM ...
-- 改后
SELECT source_id AS src_id, target_id AS tgt_id, file_path, ... FROM ...
```

**教训**:不同后端返回字段不同。NanoVectorDB 返回所有 meta_fields,PG 的 SQL 是手写的只返回固定字段。

## 五、三层防御的价值

| 层 | 作用 | 如果只有这一层 |
|---|---|---|
| workspace 隔离 | 跨部门物理隔离 | 同部门内无法做文档级权限 |
| pre-filter | LLM 看不到越权内容 | 改核心有 upstream 冲突风险 |
| post-fetch | 防御纵深兜底 | 只过滤引用,LLM 文本可能泄露 |

三层叠加 = 既有租户级隔离,又有文档级细粒度控制,还有双保险兜底。
