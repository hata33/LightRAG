# 练习 2:文档级 ACL(post-fetch)

**目标**:同一 workspace 内,alice 能看「月报」但看不到「薪资表」。

---

## 一、ReBAC 权限引擎

### 1.1 为什么用 ReBAC

| 模型 | 表达能力 | 例子 |
|---|---|---|
| RBAC | 角色→权限 | "alice 是 editor 角色" |
| **ReBAC** | 关系图→权限 | "alice 是 finance_report 的 viewer" |

ReBAC 能表达细粒度的文档级权限,基于 Google Zanzibar 论文。

### 1.2 Schema 定义

```zed
definition user {}

definition document {
    relation viewer: user
    relation owner: user
    permission view = viewer + owner
}
```

### 1.3 纯 Python 实现(接口兼容 SpiceDB)

原计划用 Docker 跑 SpiceDB,镜像拉取超时。改用纯 Python 实现,API 与 authzed-py 一致:

```python
class PermissionEngine:
    def write_relationship(self, rtype, rid, relation, stype, sid):
        """写关系:document:report#viewer@user:alice"""
        self._relationships.add((rtype, rid, relation, stype, sid))

    def check_permission(self, rtype, rid, permission, stype, sid) -> bool:
        """检查:user 能否 view document"""
        # permission "view" = viewer + owner(并集)
        relations = self._schema[rtype]["permissions"][permission]
        return any(self._has_direct_relation(...) for rel in relations)

    def lookup_resources(self, rtype, permission, stype, sid) -> list[str]:
        """查:user 能看的所有文档(用于 pre-filter 白名单)"""
```

### 1.4 预置权限关系

| 用户 | 文档 | 关系 | 效果 |
|---|---|---|---|
| alice | finance_report | viewer + owner | ✅ 能查看 |
| alice | salary_table | (无) | ❌ 无法查看 |

## 二、Post-fetch 过滤的工作原理

```
1. 客户端: POST /query { "query": "CEO年薪多少" }
2. 代理:   验 JWT → user_id=alice → workspace=tenant_finance → port 9621
3. 代理:   转发给后端
4. 后端:   向量检索 + LLM 生成 → 返回 { response, references }
5. 代理:   【ACL 过滤】对 references 里每个 file_path:
             salary_table.txt → doc_id=salary_table → can_view(alice, salary_table) = False → 移除
             finance_report.txt → can_view = True → 保留
6. 代理:   返回 { response, references(过滤后), _acl_denied_documents }
```

## 三、验证结果

```
alice 查 "CEO年薪" →
  回答: "CEO张三的年薪是850万"        ← ⚠️ 薪资数字泄露到文本!
  引用: [finance_report.txt]          ← salary_table 被过滤了
  被拒: [salary_table, finance_report_2025]
```

## 四、⚠️ Post-fetch 过滤的局限性(关键发现)

**代理过滤了引用列表,但 LLM 在生成回答时已经看到了越权文档的内容。**

```
问题:
  代理在后端返回结果后才过滤(第5步)
  但后端的 LLM 在生成答案时(第4步),上下文已包含 salary_table 的 chunk
  生成的文本就带着薪资数字 "850万"
  代理过滤的只是返回给客户端的 references JSON
```

**为什么**:代理层只能控制「返回什么」,不能控制「LLM 生成时看到了什么」。

**怎么彻底解决**:需要在**检索阶段**就排除越权文档 → 练习 3 的向量层 pre-filter。

---

## 五、学到的核心概念

1. **ReBAC 权限模型**:Google Zanzibar 的核心概念(relationship → permission)
2. **文档级 ACL**:同一 workspace 内不同文档不同权限
3. **Post-fetch 过滤**:在后端返回后过滤,实现简单但有根本局限
4. **file_path → doc_id 映射**:LightRAG 用文件路径,SpiceDB 用对象 ID
