## 二、Faithfulness（忠实度）

### 2.1 含义

> 答案中的每个陈述，是否都能在检索到的上下文中找到支持？**检测幻觉。**

### 2.2 打分原理

```mermaid
graph LR
    ANSWER["LLM 答案<br/>━━━━━━━━━<br/>'张三是A公司CEO，<br/>2024年薪资500万'"]

    ANSWER --> SPLIT["拆分为独立陈述<br/>━━━━━━━━━<br/>S1: 张三是A公司CEO<br/>S2: 2024年薪资500万"]

    SPLIT --> CHECK1{"S1 能在<br/>context 中<br/>找到支持?"}
    SPLIT --> CHECK2{"S2 能在<br/>context 中<br/>找到支持?"}

    CHECK1 -->|"是"| HIT1["✅ 支持"]
    CHECK2 -->|"否"| MISS2["❌ 无支持（幻觉）"]

    HIT1 --> CALC["Faithfulness<br/>= 支持的陈述数 / 总陈述数<br/>= 1/2 = 0.5"]

    style MISS2 fill:#ffcdd2
    style CALC fill:#fff3e0
```

### 2.3 分值解读

| 分值 | 含义 | 动作 |
|---|---|---|
| >0.80 | 答案忠实于检索内容 | ✅ |
| 0.60-0.80 | 有少量编造 | 检查 LLM prompt |
| <0.60 | 大量幻觉 | 换更强模型 / 加约束 |

### 2.4 低分根因与优化

| 根因 | 优化 |
|---|---|
| LLM 喜欢编造 | 在 user_prompt 加「只基于检索内容回答，不知道就说不知道」 |
| 检索的上下文质量差 | 改善召回（见 Context Recall） |
| 模型能力不足 | 换更强 QUERY 模型 |
| 上下文太少 | 增大 `max_total_tokens` |

## 三、Answer Relevance（切题度）

### 3.1 含义

> 答案是否真正回答了用户的问题？**检测跑题。**

### 3.2 打分原理

```mermaid
graph LR
    Q["用户问题<br/>━━━━━━━━━<br/>'养老保险申领条件?'"]
    A["LLM 答案<br/>━━━━━━━━━<br/>'养老保险是一种<br/>社会保险制度...'"]

    A --> REVERSE["RAGAS 反向生成问题<br/>━━━━━━━━━<br/>从答案反推:<br/>'养老保险是什么?'<br/>'养老保险有哪些类型?'"]

    REVERSE --> COMPARE["与原问题比较<br/>━━━━━━━━━<br/>原问题: '申领条件?'<br/>反推问题: '是什么?'<br/>→ 相关性低"]

    COMPARE --> SCORE["Answer Relevance = 低分<br/>答案跑题了"]

    style SCORE fill:#ffcdd2
```

> RAGAS 的巧妙设计：不直接判断「答案是否切题」，而是**从答案反生成问题**，看反生成的问题和原问题有多相似。

### 3.3 低分根因与优化

| 根因 | 优化 |
|---|---|
| 检索到的内容和问题不相关 | 调高 `COSINE_THRESHOLD`（过滤低质量召回） |
| LLM 理解错了问题方向 | 在 user_prompt 明确回答方向 |
| mode 选错 | 简单事实用 local，综述用 global，通用用 mix |
| 上下文噪声干扰 | 开启 Rerank |

## 四、Context Recall（上下文召回）

### 4.1 含义

> 标准答案中的信息，是否都被检索到的上下文覆盖了？**检测漏检。**

### 4.2 打分原理

```mermaid
graph LR
    TRUTH["标准答案<br/>━━━━━━━━━<br/>'条件: 60岁+15年<br/>材料: 身份证+户口本<br/>地点: 户籍地社保局'"]

    CONTEXT["检索到的上下文<br/>━━━━━━━━━<br/>'条件: 60岁+15年 ✅<br/>材料: 身份证 ✅<br/>地点: (没提到) ❌'"]

    TRUTH --> COMPARE["逐条比对"]
    CONTEXT --> COMPARE

    COMPARE --> SCORE["Recall = 被覆盖的条数 / 标准答案总条数<br/>= 2/3 = 0.67<br/>'地点'信息没召回"]

    style SCORE fill:#fff3e0
```

### 4.3 低分根因与优化

| 根因 | 优化 |
|---|---|
| 召回数量不够 | 增大 `TOP_K` / `CHUNK_TOP_K` |
| Embedding 质量差 | 换更好的 embedding 模型 |
| 分块太大/太小 | 调 `CHUNK_SIZE`（太大信息分散，太小上下文不足） |
| 文档解析丢内容 | 换 native/mineru 引擎 |
| 关键词没抽准 | 换更强的 KEYWORDS 模型 |
| mode 选错 | mix 三路全开覆盖最广 |

### 4.4 这是最重要的指标

> Context Recall 低 = **检索端出了问题**——相关信息压根没召回。后面生成再好也补不回来。**优先优化这个指标。**

## 五、Context Precision（上下文精度）

### 5.1 含义

> 检索到的上下文，有多少是真正相关的？**检测噪声。**

### 5.2 打分原理

```mermaid
graph TD
    CONTEXT["检索到的5个chunk<br/>━━━━━━━━━<br/>chunk-1: ✅ 高度相关<br/>chunk-2: ❌ 无关内容<br/>chunk-3: ✅ 相关<br/>chunk-4: ❌ 无关<br/>chunk-5: ✅ 相关"]

    CONTEXT --> CHECK["RAGAS 逐条评估<br/>━━━━━━━━━<br/>每个chunk: 相关 or 无关?<br/>（每条调1次LLM）"]

    CHECK --> SCORE["Precision = 相关数 / 总数<br/>= 3/5 = 0.60<br/>有40%的噪声"]

    style SCORE fill:#fff3e0
```

> **注意**：Context Precision 对每个检索到的 chunk 调一次 LLM 判断相关性。`EVAL_QUERY_TOP_K=10` 意味着每题 10 次 LLM 调用，容易触发限流。

### 5.3 低分根因与优化

| 根因 | 优化 |
|---|---|
| 召回太多无关内容 | 降低 `TOP_K` |
| chunk 太大混入无关信息 | 减小 `CHUNK_SIZE` |
| 缺少重排 | **开启 Rerank**（`enable_rerank=true`） |
| cosine 阈值太低 | 调高 `COSINE_THRESHOLD` |

### 5.4 Rerank 是最有效的优化

```
召回 60 个 → Rerank 打分排序 → 只保留最相关的 20 个
                                    ↑
                              Precision 大幅提升
```

## 六、四指标的关系

```mermaid
graph TD
    subgraph 检索端["检索端问题"]
        CR["Context Recall 低<br/>━━━━━━━<br/>信息没召回"]
        CP["Context Precision 低<br/>━━━━━━━<br/>召回有噪声"]
    end

    subgraph 生成端["生成端问题"]
        F["Faithfulness 低<br/>━━━━━━━<br/>答案有幻觉"]
        AR["Answer Relevance 低<br/>━━━━━━━<br/>答案跑题"]
    end

    CR -.->|"信息不全<br/>导致答案不全"| AR
    CP -.->|"噪声干扰<br/>导致答案偏离"| F
    CP -.->|"噪声干扰<br/>导致跑题"| AR

    style 检索端 fill:#e3f2fd
    style 生成端 fill:#fff3e0
```

**因果链**：
- Recall 低（没召回）→ 答案信息不全 → Relevance 可能低
- Precision 低（有噪声）→ 干扰生成 → Faithfulness/Relevance 可能低
- **检索端是根因，生成端是表现**

## 七、优化优先级

```mermaid
graph TD
    P1["第1优先: Context Recall<br/>━━━━━━━━━━━<br/>信息没召回=后面全白搭<br/>优化: 增大TOP_K/换embedding/改分块"]

    P2["第2优先: Context Precision<br/>━━━━━━━━━━━<br/>噪声干扰生成质量<br/>优化: 开rerank/降TOP_K"]

    P3["第3优先: Faithfulness<br/>━━━━━━━━━━━<br/>幻觉影响可信度<br/>优化: 加约束/换模型"]

    P4["第4优先: Answer Relevance<br/>━━━━━━━━━━━<br/>跑题影响体验<br/>优化: 调mode/改prompt"]

    P1 --> P2 --> P3 --> P4

    style P1 fill:#ffcdd2,stroke:#c62828
    style P2 fill:#fff3e0,stroke:#e65100
    style P3 fill:#e8f5e9,stroke:#2e7d32
    style P4 fill:#e3f2fd,stroke:#1565c0
```

> **先修检索（Recall→Precision），再修生成（Faithfulness→Relevance）**。因为检索是基础，检索不到的信息生成端怎么也补不出来。

## 八、完整优化对照表

| 指标低 | 检索端还是生成端？ | 具体优化 | 对应参数 |
|---|---|---|---|
| Context Recall 低 | 检索端 | 增大召回 | `TOP_K=80` |
| | | 换 embedding | `EMBEDDING_MODEL` |
| | | 改善分块 | `CHUNK_SIZE` / 策略R/V/P |
| | | 三路全开 | `mode=mix` |
| Context Precision 低 | 检索端 | 开 rerank | `RERANK_BINDING=cohere` |
| | | 降低召回量 | `TOP_K=30` |
| | | 提高阈值 | `COSINE_THRESHOLD=0.3` |
| Faithfulness 低 | 生成端 | 加约束 prompt | `user_prompt="只基于检索内容"` |
| | | 增加上下文 | `MAX_TOTAL_TOKENS=30000` |
| | | 换更强模型 | `LLM_MODEL=gpt-4o` |
| Answer Relevance 低 | 混合 | 调 mode | `mode=mix` |
| | | 加方向 prompt | `user_prompt` |
| | | 调高阈值 | `COSINE_THRESHOLD` |
