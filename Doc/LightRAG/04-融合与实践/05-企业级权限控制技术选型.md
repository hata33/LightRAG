## 一、现状评估：LightRAG 离企业级还差多远

LightRAG 的**检索内核**（graph RAG、多模式查询、pipeline 并发契约）工程质量很高，但**外围**（身份、多租户、审计、可观测性、水平扩展、合规）几乎为零。工作量分配大约是：内核 20%，外围 80%。

### 1.1 八个维度的差距

| 维度 | 现状 | 差距 | 优先级 |
|---|---|---|---|
| **授权（RBAC）** | JWT 里有 `role` 字段，但**全项目无一处代码读取它做权限判断**。所有认证用户拥有完全相同权限 | 无角色分级、无文档级权限 | 🔴 P0 |
| **多租户** | `workspace` 概念存在（`base.py:163`），但 HTTP 头 `LIGHTRAG-WORKSPACE` 只用于查 `pipeline_status`，**不影响实际数据路由**。整个 API 共享一个启动时创建的 LightRAG 实例 | 假多租户，无身份↔workspace 绑定 | 🔴 P0 |
| **审计日志** | `grep audit` 只返回 chunker 的 "audit mode"，无任何「谁在何时访问/修改了什么」的记录 | 完全缺失（SOX/HIPAA/GDPR 硬性要求） | 🟠 P1 |
| **可观测性** | 无 `/metrics`、无 Prometheus、无 OpenTelemetry、无 request ID、无结构化日志 | 只有文件日志 + liveness 健康检查 | 🟠 P1 |
| **数据治理** | 删除是物理硬删，无软删除、无版本、无回收站、无保留策略、无 PII 检测 | 手动备份，无生命周期管理 | 🟡 P2 |
| **合规（GDPR）** | 零代码涉及，无加密 at rest（MD5 仅用于内容 hash，非安全用途） | 无数据驻留、无被遗忘权支持 | 🟡 P2 |
| **速率限制** | 无 `slowapi`、无 `Limiter`，`/login` 可被暴力破解，`/query` 无限流 | 完全缺失 | 🔴 P0 |
| **水平扩展** | `shared_storage.py` 用 `multiprocessing.Manager` + `mp.Lock`，**只能在单机多 worker 间工作，跨机器直接失效** | 架构性阻塞，需分布式协调层 | 🔵 P3 |

### 1.2 已有的「好底子」

并非全是坏消息，以下部分工程质量不错，可以复用：

- **SQL 注入防护**：Postgres 后端全程 asyncpg 参数化查询（`$1`, `$2`, `ANY($2)`），标识符来自固定 namespace 集合
- **文件上传安全**：`MAX_UPLOAD_SIZE`（默认 100MB）+ 文件名路径穿越防护（`sanitize_filename`）+ 扩展名白名单
- **JWT 认证**：HS256 可配置，显式拒绝 `none` 算法，支持 bcrypt 密码、token 自动续期（滑动窗口）
- **workspace 隔离原语**：虽然未完整接线，但存储后端已支持 workspace 列过滤、collection 前缀、payload 分区
- **pipeline 并发契约**：`busy` / `destructive_busy` / `scanning_exclusive` 互斥设计（见 `AGENTS.md` 那张表），单机内很健壮

## 二、Graph RAG 权限为什么特别难

### 2.1 与传统 RAG 的对比

```
传统 RAG（朴素向量检索）：
  query → 向量搜索 chunk → 返回
  权限只需：在向量搜索时按 doc_id 过滤即可 ✅ 简单

Graph RAG：
  query → 向量搜索 entity/relation → 沿知识图谱多跳扩展
         → 用 entity/relation 反查关联 chunk → 组装上下文
  权限难点：
  1. entity/relation 是跨文档共享的 ——「苹果公司」可能来自 100 个文档
  2. 一个 relation 的 source_id 可能来自多个不同权限的文档
  3. 知识图谱的多跳扩展可能「绕过」权限边界
  4. 向量检索是「匿名的」—— 返回 top-k 不带权限标签
```

**核心矛盾**：实体/关系天然是**跨文档聚合**的，但权限是**按文档**划分的。

### 2.2 知识图谱多跳泄漏的典型场景

```
文档A（用户可见） → 实体「项目X」→ 关联实体「客户Y」（来自文档B，用户不可见）

如果只做向量搜索层的过滤：
  - 向量搜索「项目X」命中 ✅
  - _find_most_related_edges_from_entities 沿图谱找到「项目X—合作—客户Y」这条边
  - 这条边的 source_id 包含文档 B 的 chunk
  - 如果不在这层过滤，用户通过「客户Y」间接看到了文档 B 的内容 ❌ 泄漏
```

### 2.3 实体跨文档的权限语义问题

```
实体「苹果公司」同时来自：
  - 文档1（财务部，restricted）
  - 文档2（市场部，internal）
  - 文档3（公开新闻，public）

用户能看文档2和3，不能看文档1。这个实体该不该参与检索？描述该怎么组装？

策略 A（OR 语义，推荐）：任意来源可见 → 参与，但描述只含可见部分
策略 B（AND 语义，最严）：所有来源都可见 → 才参与
策略 C（描述级过滤，最精）：参与检索，但描述按来源分段，只返回可见部分
```

## 三、权限模型设计

### 3.1 三种权限粒度

| 粒度 | 说明 | 实现难度 | 适用场景 |
|---|---|---|---|
| **文档级（Doc-level）** | 文档标注可见范围（部门/团队/角色），检索时按文档过滤 | ★★★☆☆ | 最常见，80% 场景够用 |
| **实体级（Entity-level）** | 实体本身标注可见性，敏感实体不参与检索 | ★★★★☆ | 高密级实体（核心客户名单） |
| **字段/段落级（Field-level）** | 文档内某些段落脱敏或限制 | ★★★★★ | 合同金额、个人薪资等 |

**建议从文档级做起**，这是性价比最高的一层。

### 3.2 ACL 数据模型

```python
# 在 DocProcessingStatus.metadata 里加 ACL 信息（base.py:838 现有的 metadata 字段）
{
    "acl": {
        "owner": "user:zhang_san",
        "visibility": "restricted",              # public | internal | restricted | confidential
        "allowed_groups": ["dept:finance", "role:analyst"],
        "allowed_users": ["user:li_si"],
        "denied_users": [],                       # 显式拒绝（优先级最高）
        "classification": "confidential",         # 密级
    }
}
```

```python
# 用户的权限身份（从 JWT claim 里解出，放入 QueryParam）
class ACLContext:
    user_id: str                          # "user:zhang_san"
    groups: set[str]                      # {"dept:finance", "role:analyst"}
    roles: set[str]                       # {"admin", "viewer"}
    clearance_level: int                  # 密级 0=public, 1=internal, 2=confidential, 3=secret
    is_admin: bool                        # admin 角色绕过所有检查
```

### 3.3 权限判定函数

```python
def can_access(acl: dict, context: ACLContext) -> bool:
    if context.is_admin:
        return True
    if context.user_id in acl.get("denied_users", []):
        return False                       # deny 优先
    if acl.get("visibility") == "public":
        return True
    if context.user_id in acl.get("allowed_users", []):
        return True
    if context.groups & set(acl.get("allowed_groups", [])):
        return True                        # 用户组有交集
    if context.clearance_level >= acl.get("classification_level", 99):
        return True                        # 密级达标
    return False
```

### 3.4 典型企业角色权限矩阵

| 角色 | 能看到的文档范围 | 检索行为 |
|---|---|---|
| **admin** | 全部 | `is_admin=True`，跳过所有过滤 |
| **dept_manager** | 本部门所有 + 公开 | 按 `dept:xxx` group 过滤 |
| **employee** | 本部门 internal 以下 + 公开 | group + classification_level ≤ 1 |
| **auditor** | 无业务数据，只看审计日志 | `allowed_doc_set = ∅`，查询返回空 |
| **external_partner** | 仅标了 `allowed_users` 含其 id 的文档 | 显式白名单 |

## 四、六种方案全景对比

### 4.1 方案速查表

```
网关代理      Wrapper      策略引擎      多图隔离      元数据过滤    深度内核
(零侵入)     (公开API)    (Cerbos/     (每租户独立   (改向量查询)  (改operate.py)
                          SpiceDB)      LightRAG实例)
```

| 方案 | 改 LightRAG 核心 | 安全强度 | 练习价值 | 工作量 | upstream 冲突 |
|---|---|---|---|---|---|
| ① 网关代理 | ❌ 完全不改 | ★★☆☆☆ | ★★★☆☆ | 2-3天 | 零 |
| ② Wrapper 层 | ❌ 完全不改 | ★★★☆☆ | ★★★★☆ | 3-5天 | 零 |
| ③ 策略引擎 | ❌ 完全不改 | ★★★★☆ | ★★★★★ | 5-7天 | 零 |

| ⑥ 深度内核 | ✅ 大改 | ★★★★★ | ★★★★★ | 15-17天 | 极高 |

### 4.2 每种方案详解

#### 方案 ① 网关代理（Sidecar/Reverse Proxy）— 零侵入

```
                    独立进程(你的代码)
                 ┌──────────────────────┐
用户 ──► [Nginx] │  FastAPI 网关         │ ──► LightRAG API(原封不动)
                 │  - JWT 验证           │     localhost:9621
                 │  - RBAC 路由          │
                 │  - workspace 选择     │
                 │  - 限流 / 审计日志    │
                 └──────────────────────┘
```

**业界依据**：这是 Sidecar 模式的标准做法，Istio/Envoy/Kong AI Gateway 都是这个思路。

**原理**：你的网关作为一个独立 FastAPI 服务，接收所有请求，验证身份后转发给 `localhost:9621` 的 LightRAG。LightRAG 完全不知道网关的存在。

**适合场景**：想练 RBAC + JWT + 中间件，但不想碰 RAG 内部。

**局限**：权限粒度只能到 **workspace 级**(租户隔离)，做不了文档级 —— 因为网关看不到检索结果内部。

#### 方案 ② Wrapper 层（封装公开 API）— 零侵入

```
                 你的代码(一个 Python 类)
              ┌────────────────────────────────┐
你的应用 ───► │  PermissionAwareRAG            │ ──► rag.aquery_data()
              │  - 登录时构造 ACLContext        │      (LightRAG 公开 API)
              │  - 查询前算 allowed_doc_set    │
              │  - 查询后过滤 entities/chunks  │  ← post-fetch 过滤
              │  - 审计日志                    │
              └────────────────────────────────┘
```

**业界依据**：LangChain + ChromaDB + Cerbos 的 RAG 授权教程和 Databricks 的 ACL+元数据过滤方案都是这个模式 —— 用公开 API 包装，不改底层。

**关键点**：用 `aquery_data()`(返回结构化的 `entities/relationships/chunks`，每个带 `source_id` / `file_path`)而非 `aquery()`(返回纯字符串)，这样你能拿到结构化数据做过滤。

**学到的核心概念**：

- 你会**亲手验证** post-fetch 的局限性 —— LLM 上下文里已塞入越权内容，你过滤的只是返回给前端的 JSON。这种"踩坑"本身就是深刻的学习。
- 理解 graph RAG 检索结果的数据结构

**适合场景**：这是性价比最高的练习方案，既有真实的安全实现，又不碰核心。

#### 方案 ③ 外部策略引擎（SpiceDB / Cerbos）— 零侵入，学习价值最高

```
                 你的代码
              ┌──────────────────────────┐
你的应用 ───► │  PermissionAwareRAG      │
              │  1. 查询前:              │ ──► SpiceDB / Cerbos(独立服务)
              │     "用户U能看哪些文档?"  │ ◄── 返回 allowed_doc_ids[]
              │     → 拿到 doc 白名单     │
              │  2. 用白名单调 LightRAG   │ ──► rag.aquery()
              │  3. 查询后:              │
              │     "用户U能看这条结果?"  │ ──► SpiceDB 实时鉴权
              │     → 过滤不允许的        │
              └──────────────────────────┘
```

**业界依据**：这是 2025-2026 年的主流最佳实践。

- **SpiceDB 官方 RAG 方案**：Google Zanzibar 论文的开源实现，pre-filter(查前拿白名单)+ post-filter(查后实时鉴权)双保险
- **Pinecone 的 RAG 访问控制教程**：系统对比了 pre-filter 和 post-filter 两种模式
- **Cerbos + LangChain 教程**：用 Cerbos 做 RBAC，和 RAG 框架解耦

**为什么学习价值最高**：

- **ReBAC**(Relationship-Based Access Control)是 Google、Airbnb、GitHub 都在用的权限模型，远比简单 RBAC 强大
- 你学的是**可迁移的技能** —— SpiceDB/Cerbos 用在任何项目都行，不只是 LightRAG
- 策略引擎是**独立服务**，和 LightRAG 完全解耦，零冲突
- Docker 一行命令就能跑起来：`docker run -p 50051:50051 authzed/spicedb`

**适合场景**：想深入学权限系统的设计，这是最值得投入的方向。

#### 方案 ④ 多图隔离（每租户独立实例）— 仅配置层

```
租户A ──► LightRAG 实例 1 (workspace=tenant_a, 独立存储)
租户B ──► LightRAG 实例 2 (workspace=tenant_b, 独立存储)
租户C ──► LightRAG 实例 3 (workspace=tenant_c, 独立存储)
         ↑ 物理隔离,不可能串租户
```

**业界依据**：FalkorDB 的多图拓扑和 GraphRAG 多租户隔离讨论都指出——**结构隔离 > 逻辑过滤**，因为"忘了加 filter"是最常见的泄漏 bug。

**原理**：LightRAG **已经有** workspace 机制(上一轮调研确认 `base.py:163`)，每个 workspace 的数据天然隔离。你只需要：

- 维护一个 `workspace → LightRAG 实例` 的池
- 请求路由时按用户身份选实例

**优点**：**绝对安全**(物理隔离，不可能串)，实现极简。

**缺点**：**资源浪费**(每个租户一个实例)，**只做租户级**，做不了同一租户内的文档级权限。但如果是纯练习，这个"租户=部门"的粒度已经够 demo 了。

**适合场景**：想最快看到"不同角色看到不同数据"的效果，投入产出比最高。

#### 方案 ⑤ 元数据过滤（改向量查询接口）— 中度侵入

```
改 BaseVectorStorage.query() 加 filter 参数
  ├─ Qdrant: Filter(must=[doc_id ∈ allowed])
  ├─ Postgres: WHERE doc_id = ANY($5)
  └─ Milvus: expr="ARRAY_CONTAINS_ANY(doc_ids, ...)"
```

**业界依据**：Supabase RLS for RAG、Databricks Vector Search 都强调——向量数据库的元数据过滤是 RAG 权限的核心机制。OpenAI 社区讨论也确认"metadata filtering 是目前最通用的方案"。

**适合场景**：想深入学向量数据库内部机制。但要改 `base.py` 接口，有 upstream 冲突风险。

#### 方案 ⑥ 深度内核改造 — 高侵入

就是上一轮我给的那套完整方案（见第五章）。**练习场景不推荐**。

## 五、深度内核方案的三层防御架构

> ⚠️ 本章是完整方案的理论参考，**练习场景不建议直接实施**，仅供理解原理。

权限过滤必须在**多个层级**执行，任何单层都不够：

### 5.1 三层防御总览

```
┌─────────────────────────────────────────────────────────────┐
│  第 1 层：索引层（Index-time）— 写入时就打好权限标签          │
│  entity/relation/chunk 的 payload 里写入 doc_id / acl_groups │
│  → 让后续过滤「有东西可过滤」                                │
└─────────────────────────────────────────────────────────────┘
          │
          ▼
┌─────────────────────────────────────────────────────────────┐
│  第 2 层：检索层（Retrieval-time）— 尽可能在向量搜索时过滤    │
│  向量库的 payload filter 直接按 allowed_doc_ids 过滤         │
│  → 性能最好，数据库引擎原生支持                              │
└─────────────────────────────────────────────────────────────┘
          │
          ▼
┌─────────────────────────────────────────────────────────────┐
│  第 3 层：兜底层（Post-fetch）— 图谱扩展后、返回前的最终检查  │
│  对所有 entity/relation/chunk 逐一校验 source_id 是否在      │
│  allowed_doc_set 内，不在就丢弃                              │
│  → 防止知识图谱多跳「绕过」权限边界                          │
└─────────────────────────────────────────────────────────────┘
```

### 5.5 性能注意事项

**allowed_doc_set 的计算与缓存**：

```python
@cached(ttl=300, key=lambda ctx: f"allowed_docs:{ctx.user_id}:{ctx.workspace}")
async def compute_allowed_doc_set(ctx: ACLContext) -> set[str]:
    # 一次 SQL 查出该用户可见的所有 doc_id
    ...
```

**向量搜索召回不足问题**：

- top_k=60 过滤后可能只剩 10 个
- 方案：过滤前放大召回（top_k × 2 或 × 3），过滤后截断
- Qdrant/Milvus/Postgres 原生 filter 性能足够，无需放大
- FAISS/NanoVectorDB 无原生 filter，必须放大召回 + Python post-filter

## 六、推荐路线：渐进式练习方案

> 针对「fork 活跃 upstream + 练习而非生产」的场景。

### 6.1 为什么不建议直接大改核心

当前项目 fork 自 `HKUDS/LightRAG`（upstream 持续活跃，不断合入 PR）。大改核心的风险：

| 风险 | 具体表现 |
|---|---|
| **无法 sync upstream** | 上游改了 `operate.py`（检索核心），ACL 改动全在那，合并冲突极其痛苦 |
| **改出隐藏 bug** | `shared_storage.py` 的并发契约、`pipeline_status` 状态机极脆弱 |
| **投入产出不匹配** | 完整 ACL 约 17 天，但 80% 价值在前 20% 改动里 |

### 6.2 推荐组合：方案 ④ → ③ → ②（三个独立练习，递进加深）

```
练习 1(1-2天):多图隔离 —— 最快见效
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  • 用 LightRAG 现成的 workspace 机制
  • 写一个 WorkspaceManager 类,维护 workspace→RAG 实例池
  • 不同用户登录路由到不同 workspace
  • 成果:能 demo "财务部用户看不到市场部数据"
  • 学到:多租户隔离的两种模式(物理 vs 逻辑)

练习 2(3-5天):接入 SpiceDB —— 学企业级权限
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  • Docker 起一个 SpiceDB
  • 定义 schema: user → group → document 的 ReBAC 关系
  • pre-filter:查 SpiceDB 拿到用户可见的 doc_id 列表
  • 成果:文档级权限,不是租户级
  • 学到:Google Zanzibar 模型、ReBAC、pre/post-filter
  • 这是含金量最高的练习

练习 3(2-3天):Wrapper post-fetch 过滤 —— 理解局限
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  • 用 aquery_data() 拿结构化结果
  • 按 source_id/file_path 过滤
  • 亲手验证"为什么 post-fetch 不够安全"
  • 学到:安全边界的权衡、graph RAG 多跳泄漏
```

### 6.3 为什么这个组合最优

| 考量 | 这个方案的表现 |
|---|---|
| **零 upstream 冲突** | ✅ 三个练习都是纯新增文件 + 外部服务，不碰 LightRAG 核心 |
| **递进学习** | ✐ 从"物理隔离"到"逻辑过滤"到"理解局限"，难度递增 |
| **每个都能独立 demo** | ✐ 不依赖前一个练习，可单独跑通 |
| **学到的技能可迁移** | ✐ SpiceDB/ReBAC/JWT/多租户 —— 任何后端项目都用得上 |
| **符合业界最佳实践** | ✐ SpiceDB 方案就是 Google/Pinecone/Cerbos 在推的主流模式 |

### 6.4 目标架构图

```
                    你的练习项目(全新目录,如 kb_gateway/)

   ┌─ 练习1: WorkspaceManager ──────────────────────┐
   │  tenant_a用户 ──► rag_instances["tenant_a"]     │
   │  tenant_b用户 ──► rag_instances["tenant_b"]     │  ← 物理 isolation

   │         ──► [doc_1, doc_3, doc_7]  白名单       │

   └─────────────────────────────────────────────────┘
          │
          ▼
   ┌─ 练习3: Post-fetch 安全网 ─────────────────────┐
```

### 6.5 upstream 同步的实操建议

```bash

```

`Doc/` 目录就是好例子 —— 纯新增，不和 upstream 冲突，这是正确的姿势。
