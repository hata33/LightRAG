| 场景 | NetworkX（默认） | Neo4j/PG AGE |

| **推荐规模** | < 10万节点 | 百万级+ |

## 二、为什么 NetworkX 大了会慢

### 2.1 根本原因：全量序列化

```mermaid
graph TD
    subgraph NETWORKX["NetworkX 的存储模型"]
        direction TB
        MEM["内存中的 nx.Graph<br/>全部节点+边"]
        FILE["磁盘上的 GraphML<br/>单个 graph.graphml 文件"]

        MEM -->|"每次 _insert_done<br/>全量写出"| FILE
        FILE -->|"每次启动 / 检测到更新<br/>全量读入"| MEM
    end

    subgraph PROBLEM["问题"]
        P1["写: O(N+E) 全量序列化<br/>━━━━━━━━━━━━━<br/>每批入库结束都全量写<br/>10万节点=每次数秒"]
        P2["读: O(N+E) 全量加载<br/>━━━━━━━━━━━━━<br/>启动时全量读入内存<br/>10万节点=启动慢+占内存"]
        P3["内存: 全图常驻<br/>━━━━━━━━━━━━━<br/>10万节点约 1-3GB RAM"]
    end

    style PROBLEM fill:#ffcdd2,stroke:#c62828
```

源码证据（`networkx_impl.py`）：
- `write_nx_graph`（:131）：`nx.write_graphml(graph, tmp)` —— **全量写出**
- `load_nx_graph`（:125）：`nx.read_graphml(file_name)` —— **全量读入**
- 每次入库批次的 `_insert_done` 触发全量写盘

### 2.2 各操作的真实复杂度

| 操作 | 复杂度 | 说明 | 大规模影响 |
|---|---|---|---|
| `get_node(name)` | O(1) | 字典查找 | ✅ 不受规模影响 |
| `get_node_edges(name)` | O(degree) | 取该节点的边 | ✅ 只看该节点度数 |
| `get_knowledge_graph(label)` | O(BFS 到 max_depth) | 但有 **max_nodes=1000** 截断 | ✅ 有硬限制 |
| `upsert_node` | O(1) | 内存字典操作 | ✅ 快 |
| `_insert_done` 写盘 | **O(N+E) 全量** | **每次批次结束** | ⚠️ **这是瓶颈** |
| 启动加载 | **O(N+E) 全量** | 全量 read_graphml | ⚠️ 启动慢 |

> **关键洞察：查询操作本身不受总规模影响**（因为有限制和局部遍历）。瓶颈在**入库时的全量序列化**和**启动时的全量加载**。

## 三、查询侧：为什么不太受影响

### 3.1 多重规模限制

```mermaid
graph TD
    Q["查询: '张三负责什么?'"]

    Q --> L1["限制 1: 向量召回 top_k<br/>━━━━━━━━━━━━━<br/>TOP_K=60（默认）<br/>不管图有多大<br/>只召回最相似的60个实体"]

    L1 --> L2["限制 2: 图遍历 max_depth<br/>━━━━━━━━━━━━━<br/>get_knowledge_graph(max_depth=3)<br/>只遍历3层关系<br/>不碰全图"]

    L2 --> L3["限制 3: 子图 max_nodes<br/>━━━━━━━━━━━━━<br/>max_nodes=1000（默认）<br/>BFS 超过1000节点就截断<br/>is_truncated=True"]

    L3 --> L4["限制 4: token 预算<br/>━━━━━━━━━━━━━<br/>max_entity_tokens=6000<br/>max_relation_tokens=8000<br/>max_total_tokens=30000<br/>超预算就截断"]

    style L1 fill:#e3f2fd,stroke:#1565c0
    style L2 fill:#e8f5e9,stroke:#2e7d32
    style L3 fill:#fff3e0,stroke:#e65100
    style L4 fill:#f3e5f5,stroke:#6a1b9a
```

**四道关卡保证查询规模可控**：
1. 向量召回只取 top_k 个（默认 60），不管全图几百万节点
2. 图遍历最多 3 层深度
3. 子图最多 1000 节点就截断
4. 最终 token 预算截断

> 所以**查询 1 万节点和 100 万节点的图，延迟差异不大**——因为查询只碰局部子图。

### 3.2 查询的真实复杂度

```
向量召回:  O(top_k × log N)    ← 向量库 ANN 索引，与图规模无关
图遍历:    O(min(max_nodes, BFS3层))  ← 最多1000节点
反查chunk: O(命中实体数)         ← 通常几十个
```

## 四、入库侧：真正的瓶颈

### 4.1 合并过程的锁竞争

```mermaid
graph TD
    BATCH["一批 chunk 抽取完成<br/>产出多个实体/关系"]

    BATCH --> MERGE["merge_nodes_and_edges"]

    MERGE --> LOCK["keyed lock<br/>━━━━━━━━━━━━━<br/>每个实体名一把锁"]

    LOCK --> CASE{"同名实体<br/>同时被处理?"}

    CASE -->|"是"| WAIT["串行等待<br/>━━━━━━━━━━━━━<br/>第二个等第一个合并完<br/>大规模下热门实体<br/>成为瓶颈"]
    CASE -->|"否"| PARALLEL["并行处理<br/>━━━━━━━━━━━━━<br/>不同实体名不互锁"]

    style WAIT fill:#fff3e0,stroke:#e65100
```

> 大规模语料下，**高频实体**（如"中国"、"美国"、"AI"）会被大量 chunk 同时抽到，成为锁热点。但大多数实体是低频的，可以并行。

### 4.2 LLM 调用成本

```mermaid
graph LR
    subgraph 成本["入库的 LLM 调用量"]
        C1["抽取阶段<br/>━━━━━━━━━<br/>每个 chunk 1次（+gleaning轮）<br/>1000 chunk ≈ 1000-3000次"]
        C2["合并阶段<br/>━━━━━━━━━<br/>每个同名实体 1次重摘要<br/>热门实体被反复摘要"]
    end

    成本 --> SCALE["规模化策略<br/>━━━━━━━━━━━━━<br/>① LLM 缓存（命中跳过）<br/>② 降低 gleaning 轮数<br/>③ 用便宜模型抽取"]

    style 成本 fill:#fff3e0,stroke:#e65100
    style SCALE fill:#c8e6c9,stroke:#2e7d32
```

### 4.3 NetworkX 全量写盘

每次批次入库结束（`_insert_done`），NetworkX 会**全量写出 GraphML**：

```mermaid
graph LR
    BATCH_END["批次结束<br/>_insert_done"] --> WRITE["全量 write_graphml<br/>━━━━━━━━━━━━━<br/>O(N+E)"]

    WRITE --> SCALE_PROBLEM{"图规模"}
    SCALE_PROBLEM -->|"1万节点"| FAST["<1秒 ✅"]
    SCALE_PROBLEM -->|"10万节点"| SLOW["数秒 ⚠️"]
    SCALE_PROBLEM -->|"100万节点"| VERY_SLOW["几十秒+ ❌"]

    style VERY_SLOW fill:#ffcdd2,stroke:#c62828
```

## 五、瓶颈定位与解决方案

### 5.1 不同规模的推荐方案


**切换方式**（只改一个配置）：

```env

NEO4J_USERNAME=neo4j

```

> 业务代码零改动——所有图操作都走 `BaseGraphStorage` 抽象接口。

## 七、实用调参建议

### 7.1 查询参数（`QueryParam` / 环境变量）

| 参数 | 默认 | 大规模建议 | 作用 |
|---|---|---|---|
| `TOP_K` | 60 | 40-60 | 实体/关系召回数 |
| `CHUNK_TOP_K` | 12 | 8-12 | chunk 召回数 |
| `max_graph_nodes` | 1000 | 500-1000 | 子图节点上限 |
| `max_entity_tokens` | 6000 | 不变 | 实体 token 预算 |
| `max_total_tokens` | 30000 | 不变 | 总 token 预算 |

### 7.2 入库参数

| 参数 | 默认 | 大规模建议 | 作用 |
|---|---|---|---|
| `max_parallel_insert` | 3 | 5-10 | 并行入库数 |
| `MAX_GLEANING` | 1 | 0-1 | 补抽轮数（减成本） |
| `chunk_token_size` | 1200 | 1000-1500 | chunk 大小（小=更精细抽取） |

## 八、性能预估表

| 图规模 | NetworkX | Neo4j | 向量库建议 |
|---|---|---|---|
| **1千节点** | ✅ 毫秒级 | ✅ 毫秒级 | NanoVectorDB 够用 |
| **1万节点** | ✅ 查询快，写盘<1秒 | ✅ 快 | NanoVectorDB / FAISS |
| **10万节点** | ⚠️ 写盘数秒，启动慢 | ✅ 无感 | FAISS / Qdrant |
| **50万节点** | ❌ 不可用（GB级内存） | ✅ 正常 | Milvus / Qdrant |
| **100万+节点** | ❌ 不可用 | ✅ 正常 | Milvus 集群 |

## 九、关键结论

```mermaid
graph TB
    subgraph 总结["性能问题总结"]
        direction TB
        C1["✅ 查询不受规模影响<br/>━━━━━━━━━━━━━<br/>4道限制关卡<br/>(top_k/depth/nodes/tokens)<br/>只碰局部子图"]
        C2["⚠️ 入库是瓶颈<br/>━━━━━━━━━━━━━<br/>NetworkX 全量序列化<br/>+ LLM 调用成本<br/>+ 热门实体锁竞争"]
        C3["🔑 解决方案明确<br/>━━━━━━━━━━━━━<br/>换 Neo4j/PG AGE<br/>业务代码零改动"]
    end

    style C1 fill:#c8e6c9,stroke:#2e7d32
    style C2 fill:#fff3e0,stroke:#e65100
    style C3 fill:#e3f2fd,stroke:#1565c0
```

### 一句话回答

> **几万节点：NetworkX 可用但建议换 PG/Neo4j；几十万节点：必须换专业图库，但只需改一个配置（`LIGHTRAG_GRAPH_STORAGE`），业务代码零改动。查询本身不受规模影响（有多重限制），瓶颈在入库时的全量序列化和 LLM 调用成本。**
