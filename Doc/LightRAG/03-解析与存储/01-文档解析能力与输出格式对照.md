**项目**：LightRAG · **版本**：1.5.5 · **日期**：2026-07-08 · **作者**：15531

> 本文档回答两个问题：**「什么格式的文件能入库」** 与 **「每种格式解析后产出什么内容」**。所有结论均来自源码核实（`parser/legacy/extractors.py`、`parser/docx/`、`sidecar/writer.py`、`pipeline.py`）。全文使用 Mermaid 图表表达结构与流程。

### 能力速查（精简表）

| 格式 | legacy | native | mineru | docling | 推荐引擎 |
|---|:---:|:---:|:---:|:---:|---|
| **Word .docx** | ✅ 纯文本 | ✅ **Markdown+块** | ✅ | ✅ | **native** |
| **Excel .xlsx** | ✅ **多sheet文本** | ❌ | ✅ | ✅ | **legacy** |
| **PowerPoint .pptx** | ✅ **纯文本** | ❌ | ✅ | ✅ | **legacy** |
| **PDF 数字版** | ✅ 文字层 | ❌ | ✅ | ✅ | legacy 够用，mineru 更好 |
| **PDF 扫描件** | ❌ 提空 | ❌ | ✅ **OCR** | ❌ | **mineru** |
| **Markdown/文本** | ✅ 原样 | ✅ 原样+图 | — | ✅ | legacy 或 native |

### 3.1 Word (.docx) — 两种引擎差异对比

```mermaid

    end

    subgraph NATIVE_WORD["native 引擎输出（Markdown，推荐）"]
        direction TB
        N1["# 一级标题<br/>━━━━━━━━━━<br/>标题层级保留"]
```

**native 额外产出**（送入 P 分块和 VLM 分析）：
- `<base>.blocks.jsonl` — 结构化块（标题/段落/表格/图片分类）
- `<base>.blocks.assets/` — 提取出的图片文件

## 四、内容格式统一化流程

不同引擎解析后，如何统一成 LightRAG 内部格式：

```mermaid
graph LR
    subgraph 引擎产出
        LEGACY_OUT["legacy<br/>纯文本 str"]
        NATIVE_IR["native/mineru/docling<br/>IRDoc 中间表示"]
    end

    LEGACY_OUT -->|"直接返回<br/>不经 IR"| FINAL_LEGACY["ParseResult<br/>content = 纯文本<br/>blocks_path = 空 ❌<br/>parse_format = RAW"]

    NATIVE_IR --> WRITE["write_sidecar<br/>━━━━━━━━━━━<br/>IRDoc → 统一渲染"]
    WRITE --> RENDER["content = Markdown<br/>（\\n\\n.join 各 block）"]
    WRITE --> BLOCKS["blocks.jsonl"]
    WRITE --> ASSETS["blocks.assets/ 图片"]
    WRITE --> SIDE["tables/drawings/<br/>equations.json"]

    RENDER --> FINAL_IR["ParseResult<br/>content = Markdown ✅<br/>blocks_path = blocks.jsonl ✅<br/>parse_format = LIGHTRAG"]
    BLOCKS --> FINAL_IR
    SIDE --> FINAL_IR

    FINAL_LEGACY --> CHUNK["分块阶段"]
    FINAL_IR --> CHUNK

    CHUNK --> NOTE2["legacy 无 blocks_path<br/>→ P 分块降级为 R<br/>（不影响入库）"]

    style WRITE fill:#e8f5e9,stroke:#2e7d32

    style FINAL_IR fill:#c8e6c9,stroke:#2e7d32
```

## 五、PDF 图片语义化入库（完整链路）

这是最复杂的场景，需要解析 → VLM 分析 → 抽取三段协同：

```mermaid
graph TD
    START["📄 含图片的 PDF"]

    START --> STEP1["第 1 步：图片提取<br/>━━━━━━━━━━━━━<br/>解析阶段（Layer 1）"]

    STEP1 --> ENG{{"用了 mineru/docling?"}}
    ENG -->|"否（legacy/native）"| FAIL1["❌ 图片不会被提取<br/>drawings.json 不存在"]
    ENG -->|"是"| OK1["✅ 图片提取到<br/>blocks.assets/*.png<br/>drawings.json 记录元数据"]

    OK1 --> STEP2["第 2 步：VLM 语义理解<br/>━━━━━━━━━━━━━━━━<br/>分析阶段（Layer 2）<br/>analyze_multimodal()"]

    STEP2 --> CHECK_OPT{{"上传带 i 选项?<br/>process_opts.images"}}
    CHECK_OPT -->|"否"| SKIP["⏭️ 跳过 VLM<br/>analyzing_stage_skipped = True"]
    CHECK_OPT -->|"是"| CHECK_VLM{{"VLM 角色已配置?<br/>VLM_PROCESS_ENABLE<br/>+ vlm 模型"}}

    CHECK_VLM -->|"否"| ERR["❌ MultimodalAnalysisError<br/>VLM role is not available"]
    CHECK_VLM -->|"是"| ANALYZE["_analyze_drawing()<br/>━━━━━━━━━━━━━<br/>① 找图片路径<br/>② 检查尺寸（太小跳过）<br/>③ 检查格式（png/jpg/gif/webp）<br/>④ 调 VLM 生成语义描述（JSON）<br/>⑤ 描述注入 content"]

    ANALYZE --> STEP3["第 3 步：作为实体入库<br/>━━━━━━━━━━━━━━━━<br/>抽取阶段（Layer 3）<br/>（和普通文本完全一样）"]

    STEP3 --> CHUNK["分块 → extract_entities → 图谱"]
    CHUNK --> KG["🖼️ 图片里的内容<br/>变成知识图谱实体/关系"]

    style START fill:#fffde7,stroke:#f57f17
    style ANALYZE fill:#c8e6c9,stroke:#2e7d32
    style KG fill:#e1bee7,stroke:#6a1b9a
    style FAIL1 fill:#ffcdd2,stroke:#c62828
    style ERR fill:#ffcdd2,stroke:#c62828
    style SKIP fill:#fff3e0,stroke:#e65100
```

### 三个前提条件（缺一不可）

```mermaid
graph LR
    COND1["① 引擎能提图<br/>LIGHTRAG_PARSER=*:mineru-iteP<br/>mineru 或 docling"]

    COND3["③ VLM 角色可用<br/>VLM_PROCESS_ENABLE=true<br/>+ 多模态模型<br/>pipeline.py:3647"]

    COND1 --> ALL{"三者齐全?"}
    COND2 --> ALL
    COND3 --> ALL

    ALL -->|"是"| SUCCESS["✅ 图片语义化入库"]
    ALL -->|"缺任一"| FAIL["❌ 图片不会进图谱"]

    style SUCCESS fill:#c8e6c9,stroke:#2e7d32
    style FAIL fill:#ffcdd2,stroke:#c62828
```

### 场景 A：只用本地引擎（零外部依赖）

```env
# 默认配置即可，Excel/PPT/纯文本 Word/PDF文字层 都能处理
LIGHTRAG_KV_STORAGE=JsonKVStorage
# （其他默认）
```

### 场景 B：处理扫描件/复杂 PDF（需 mineru）

```env
MINERU_API_MODE=local
MINERU_LOCAL_ENDPOINT=http://127.0.0.1:8000
MINERU_LOCAL_BACKEND=pipeline         # CPU 友好（含 PaddleOCR）
MINERU_LOCAL_PARSE_METHOD=ocr         # 强制 OCR（扫描件必备）
MINERU_ENABLE_TABLE=true
MINERU_ENABLE_FORMULA=true
# 路由：PDF 用 mineru，其他降级 legacy
LIGHTRAG_PARSER=*.pdf:mineru-iteP;*:legacy-R
```

### 场景 C：PDF 图片语义化（全套）

```env
# 在场景 B 基础上加：
VLM_PROCESS_ENABLE=true
VLM_BINDING=openai
VLM_MODEL=gpt-4o                       # 必须多模态模型
# 上传时带 i 选项（已在 mineru-iteP 的 i 里）
```

