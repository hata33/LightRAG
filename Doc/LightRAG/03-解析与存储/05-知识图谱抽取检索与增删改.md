## 一、核心澄清：先纠正一个误解

```mermaid
graph LR
    subgraph 误解["❌ 常见误解"]
        W1["向量库相似<br/>自动构成图谱"]
        W1 --> W2["相似度高 = 有关系"]
    end

    subgraph 真相["✅ LightRAG 的实际做法"]
        R1["LLM 显式抽取<br/>实体 + 关系"]
        R1 --> R2["关系是语义判断<br/>不是向量相似度"]
        R2 --> R3["图谱是独立结构<br/>与向量库分开存"]
    end

    style 误解 fill:#ffcdd2,stroke:#c62828
    style 真相 fill:#c8e6c9,stroke:#2e7d32
```

**LightRAG 的知识图谱不是「向量相似自动连边」**，而是 **LLM 读文本后显式判断「A 和 B 有什么关系」**，把关系作为结构化数据存进图数据库。向量库只负责召回，图谱负责关系推理。

## 二、抽取：LLM 怎么生成知识图谱

### 2.1 抽取过程（`operate.py:3320 extract_entities`）

```mermaid
graph TD
    CHUNK["一个文本 chunk<br/>━━━━━━━━━━━━━<br/>例: '张三在A公司任CEO,<br/>李四是他的副手'"]

    CHUNK --> PROMPT["构造抽取 Prompt<br/>━━━━━━━━━━━━━<br/>PROMPTS entity_extraction<br/>告诉 LLM:<br/>① 找实体(人/组织/概念)<br/>② 找实体间关系<br/>③ 输出 JSON 格式"]

    PROMPT --> LLM["EXTRACT 角色 LLM<br/>━━━━━━━━━━━━━<br/>gpt-4o-mini 等<br/>带 LLM 缓存查询"]

    LLM --> PARSE["解析 LLM 输出<br/>━━━━━━━━━━━━━<br/>_process_json_extraction_result"]

    PARSE --> ENTITIES["实体列表<br/>━━━━━━━━━━━━━<br/>{<br/>  entity_name: '张三',<br/>  entity_type: 'PERSON',<br/>  description: 'A公司CEO',<br/>  source_id: 'chunk-xxx'<br/>}"]

    PARSE --> RELATIONS["关系列表<br/>━━━━━━━━━━━━━<br/>{<br/>  src_id: '张三',<br/>  tgt_id: '李四',<br/>  description: '上下级关系',<br/>  weight: 非负数,<br/>  source_id: 'chunk-xxx'<br/>}"]

    PARSE --> GLEANING{"gleaning<br/>补抽轮?"}
    GLEANING -->|"未达 MAX_GLEANING"| PROMPT
    GLEANING -->|"够了"| DONE["抽取完成"]

    style CHUNK fill:#fffde7,stroke:#f57f17
    style LLM fill:#fff3e0,stroke:#e65100
    style ENTITIES fill:#e3f2fd,stroke:#1565c0
    style RELATIONS fill:#e8f5e9,stroke:#2e7d32
```

### 2.2 是三元组吗？

**接近三元组，但更丰富。** 传统三元组是 `(主语, 谓词, 宾语)`，LightRAG 的结构是：

| 元素 | 传统三元组 | LightRAG 实体 | LightRAG 关系 |
|---|---|---|---|
| 核心字段 | subject | entity_name | src_id → tgt_id |
| 谓词 | predicate | — | description（自然语言描述） |
| 宾语 | object | — | — |
| 额外信息 | 无 | entity_type、description、source_id | weight、source_id |

```mermaid
graph LR
    subgraph 传统三元组["传统三元组"]
        T["张三 —任职→ A公司"]
    end

    subgraph LIGHTRAG["LightRAG 结构（更丰富）"]
        E["实体: 张三<br/>type: PERSON<br/>desc: A公司CEO<br/>source: chunk-1"]
        R["关系: 张三→李四<br/>desc: 上下级<br/>weight: 8.0<br/>source: chunk-1"]
        E2["实体: 李四<br/>type: PERSON<br/>desc: 副CEO<br/>source: chunk-1"]
    end

    E --- R --- E2

    style 传统三元组 fill:#eceff1
    style LIGHTRAG fill:#c8e6c9
```

> **关键区别**：关系的「谓词」是**自然语言描述**（"上下级关系"），而非固定谓词（"管理"）。这让关系表达更灵活，也意味着关系需要 embedding 来做语义召回。

## 三、存储：知识图谱存在哪

### 3.1 三个地方同时存（不是只有 Neo4j）

```mermaid
graph TB
    subgraph 抽取产物["extract_entities 产出"]
        ENT["实体列表"]
        REL["关系列表"]
    end

    subgraph 存储1["① 图数据库<br/>chunk_entity_relation_graph"]
        G["节点 + 边<br/>━━━━━━━━━━━━━<br/>节点: {name, type, desc, source_id}<br/>边: {src→tgt, desc, weight, source_id}<br/>━━━━━━━━━━━━━<br/>用于: 图遍历(多跳推理)"]
    end

    subgraph 存储2["② 实体向量库<br/>entities_vdb"]
        EV["entity_name + description 的 embedding<br/>━━━━━━━━━━━━━<br/>用于: 语义召回相关实体"]
    end

    subgraph 存储3["③ 关系向量库<br/>relationships_vdb"]
        RV["关系描述的 embedding<br/>━━━━━━━━━━━━━<br/>用于: 语义召回相关关系"]
    end

    ENT --> 存储1 & 存储2
    REL --> 存储1 & 存储3

    style 存储1 fill:#e8f5e9,stroke:#2e7d32
    style 存储2 fill:#e3f2fd,stroke:#1565c0
    style 存储3 fill:#fff3e0,stroke:#e65100
```

### 3.2 图数据库后端选择

**不一定用 Neo4j**，取决于配置：

| 后端 | 配置值 | 适用场景 |
|---|---|---|
| **NetworkX**（默认） | `graph_storage=NetworkXStorage` | 单机、内存、零依赖 |
| **Neo4j** | `graph_storage=Neo4JStorage` | 大规模图谱、专业图查询 |
| **PostgreSQL AGE** | `graph_storage=PGGraphStorage` | 生产统一存储 |
| **Memgraph** | `graph_storage=MemgraphStorage` | 高性能图查询 |
| **MongoDB** | `graph_storage=MongoGraphStorage` | 统一文档存储 |

```mermaid
graph LR
    CODE["业务代码<br/>━━━━━━━<br/>graph.get_node<br/>graph.upsert_node<br/>graph.get_node_edges<br/>graph.get_knowledge_graph"]

    CODE --> ABSTRACT["BaseGraphStorage<br/>抽象接口"]

    ABSTRACT --> NX["NetworkX<br/>内存图<br/>默认"]
    ABSTRACT --> NEO["Neo4j<br/>专业图库"]
    ABSTRACT --> PG["PG + AGE<br/>统一存储"]
    ABSTRACT --> MONGO["MongoDB"]

    style ABSTRACT fill:#ffcdd2,stroke:#c62828
    style NX fill:#e8f5e9,stroke:#2e7d32
```

> 业务代码只调抽象方法（`get_node` / `upsert_edge`），完全不感知底层是 NetworkX 还是 Neo4j。

### 3.3 节点和边的结构

```mermaid
graph LR
    subgraph 节点["图节点（upsert_node）"]
        N["entity_name: '张三'（节点ID=名称）<br/>━━━━━━━━━━━━━━━━━━<br/>entity_type: 'PERSON'<br/>description: 'A公司CEO，负责...'<br/>source_id: 'chunk-1⟨SEP⟩chunk-5'  ← 来源切片<br/>file_path: 'doc1.pdf⟨SEP⟩doc3.pdf'"]
    end

    subgraph 边["图边（upsert_edge）"]
        E["src='张三' → tgt='李四'<br/>━━━━━━━━━━━━━━━━━━<br/>description: '上下级关系，张三是李四的上级'<br/>weight: 8.0<br/>source_id: 'chunk-1'<br/>file_path: 'doc1.pdf'"]
    end

    style 节点 fill:#e3f2fd,stroke:#1565c0
    style 边 fill:#fff3e0,stroke:#e65100
```

> **`source_id` 是溯源关键**：记录这个实体/关系来自哪些 chunk，用 `⟨SEP⟩` 分隔多个来源。

## 四、跨文档关系：新文档怎么和旧文档关联

**这是最核心的问题**——LightRAG 不是每个文档建独立图，而是**所有文档共享一张图**，同名实体自动合并。

### 4.1 合并逻辑（`operate.py:2000 _merge_nodes_then_upsert`）

```mermaid
graph TD
    NEW["新文档抽取到实体 '张三'<br/>desc: 'A公司CEO'<br/>source: chunk-5"]

    NEW --> LOOKUP{"图谱中已有<br/>'张三'?"}

    LOOKUP -->|"没有"| CREATE["创建新节点<br/>━━━━━━━━━━━━━<br/>graph.upsert_node('张三', {desc, source})<br/>entities_vdb.upsert(embedding)"]

    LOOKUP -->|"已有"| MERGE["合并流程<br/>━━━━━━━━━━━━━<br/>① 取旧节点数据<br/>② 合并 source_id（旧chunk + 新chunk）<br/>③ 合并 description 列表<br/>④ LLM 重新生成摘要描述<br/>⑤ 重新 embedding<br/>⑥ upsert 更新节点"]

    MERGE --> UPDATED["合并后节点<br/>━━━━━━━━━━━━━<br/>desc: LLM合并的新描述<br/>source_id: chunk-1⟨SEP⟩chunk-5<br/>→ 两份文档都指向它"]

    style NEW fill:#fffde7,stroke:#f57f17
    style MERGE fill:#fff3e0,stroke:#e65100
    style UPDATED fill:#c8e6c9,stroke:#2e7d32
```

### 4.2 合并的细节步骤

```mermaid
graph TD
    START["合并实体 '张三'"]

    START --> S1["① 读取旧节点<br/>━━━━━━━━━━━━━<br/>graph.get_node('张三')<br/>取旧 desc / source_id / type"]

    S1 --> S2["② 合并 source_id<br/>━━━━━━━━━━━━━<br/>旧的: chunk-1, chunk-3<br/>新的: chunk-5<br/>合并: chunk-1, chunk-3, chunk-5"]

    S2 --> S3["③ 合并 description<br/>━━━━━━━━━━━━━<br/>旧desc + 新desc 拼成列表<br/>按时间戳排序"]

    S3 --> S4["④ LLM 重新摘要<br/>━━━━━━━━━━━━━<br/>_handle_entity_relation_summary()<br/>LLM 把多个描述合成一段<br/>━━━━━━━━━━━━━<br/>例: 'A公司CEO(来自doc1) +<br/>B公司顾问(来自doc3)'<br/>→ 'A公司CEO兼B公司顾问'"]

    S4 --> S5["⑤ entity_type 投票<br/>━━━━━━━━━━━━━<br/>多来源 type 用 Counter<br/>取出现最多的"]

    S5 --> S6["⑥ 重新 embedding<br/>━━━━━━━━━━━━━<br/>新描述 → embedding_func<br/>→ entities_vdb.upsert"]

    S6 --> S7["⑦ upsert 更新图节点<br/>━━━━━━━━━━━━━<br/>graph.upsert_node('张三',<br/>合并后数据)"]

    style S4 fill:#fff3e0,stroke:#e65100
    style S6 fill:#f3e5f5,stroke:#6a1b9a
```

### 4.3 跨文档关系怎么建立

```mermaid
graph LR
    subgraph DOC1["文档1: doc1"]
        D1_E1["张三"]
        D1_E2["A公司"]
        D1_R["张三→A公司: 任职CEO"]
    end

    subgraph DOC2["文档2: doc2"]
        D2_E1["李四"]
        D2_E2["A公司"]
        D2_R1["李四→A公司: 任职"]
        D2_R2["张三→李四: 上下级"]
    end

    D1_E2 -.->|"同名合并<br/>同一节点"| D2_E2
    D1_E1 -.->|"被引用<br/>同名合并"| D2_R2

    subgraph MERGED["合并后的图（全局唯一）"]
        M1["张三<br/>source: chunk1+chunk3"]
        M2["A公司<br/>source: chunk1+chunk2"]
        M3["李四<br/>source: chunk2"]
        M1 -->|"任职CEO"| M2
        M3 -->|"任职"| M2
        M1 -->|"上下级"| M3
    end

    style MERGED fill:#c8e6c9,stroke:#2e7d32
```

> **跨文档关系靠「同名实体合并」建立**：两份文档都提到「A公司」，这个节点会自动合并，`source_id` 累积两个来源。这样 doc1 的「张三→A公司」和 doc2 的「李四→A公司」就通过共享的「A公司」节点关联起来了。

### 4.4 并发安全：keyed lock

```mermaid
graph TD
    PROBLEM["并发问题<br/>━━━━━━━━━━━━━<br/>两个 chunk 同时抽到<br/>'张三' 怎么办?"]

    PROBLEM --> SOLUTION["keyed lock<br/>━━━━━━━━━━━━━<br/>每个实体名一把锁<br/>get_storage_keyed_lock(['张三'])"]

    SOLUTION --> EFFECT["效果<br/>━━━━━━━━━━━━━<br/>① '张三' 的合并串行（防竞争）<br/>② '李四' 的合并可并行（不互锁）<br/>③ 大规模并发不死锁"]

```

## 五、检索：怎么用图谱回答问题

### 5.1 检索不是「查图」，是「向量召回 + 图遍历」组合

```mermaid
graph TD
    Q["查询: '张三负责什么?'"]

    Q --> KW["关键词抽取<br/>━━━━━━━━━━━━━<br/>KEYWORDS 角色 LLM<br/>→ low_level: ['张三']<br/>→ high_level: ['职责','管理']"]

    KW --> EMBED["关键词向量化<br/>━━━━━━━━━━━━━<br/>ll_keywords → embedding<br/>hl_keywords → embedding"]

    EMBED --> VDB_Q["向量库召回<br/>━━━━━━━━━━━━━<br/>━━━ 这是第一步，不是图查询 ━━━<br/>entities_vdb.query(张三 embedding)<br/>→ 找到最相似的实体节点<br/>relationships_vdb.query(职责 embedding)<br/>→ 找到最相似的关系"]

    VDB_Q --> GRAPH_Q["图遍历（第二步）<br/>━━━━━━━━━━━━━<br/>graph.get_node_edges('张三')<br/>→ 找张三的所有关联边<br/>→ 发现: 张三→A公司(CEO), 张三→李四(上级)<br/>→ 这是图推理，向量做不到"]

    GRAPH_Q --> CHUNK_Q["反查切片（第三步）<br/>━━━━━━━━━━━━━<br/>通过 source_id 反查<br/>→ chunk-1 原文: '张三在A公司...'"]

    CHUNK_Q --> CTX["组装上下文<br/>━━━━━━━━━━━━━<br/>实体描述 + 关系描述 + chunk全文<br/>按 token 预算截断"]

    CTX --> LLM["QUERY 角色 LLM<br/>━━━━━━━━━━━━━<br/>基于上下文生成答案"]

    style VDB_Q fill:#e3f2fd,stroke:#1565c0
    style GRAPH_Q fill:#e8f5e9,stroke:#2e7d32
    style CHUNK_Q fill:#fff3e0,stroke:#e65100
```

### 5.2 向量库和图谱的分工

```mermaid
graph LR
    subgraph 向量库的职责["向量库（召回入口）"]
        V1["entities_vdb<br/>━━━━━━━<br/>把查询向量化<br/>找最相似的实体"]
        V2["relationships_vdb<br/>━━━━━━━<br/>找最相似的关系"]
        V3["chunks_vdb<br/>━━━━━━━<br/>找最相似的文本块"]
    end

    subgraph 图谱的职责["图谱（关系推理）"]
        G1["get_node_edges<br/>━━━━━━━<br/>找到张三的所有边<br/>→ 多跳推理"]
        G2["get_knowledge_graph<br/>━━━━━━━<br/>取子图<br/>→ 关系网络"]
    end

    向量库的职责 -->|"召回实体名后<br/>去图里遍历"| 图谱的职责

    style 向量库的职责 fill:#e3f2fd,stroke:#1565c0
    style 图谱的职责 fill:#e8f5e9,stroke:#2e7d32
```

> **向量库 = 找入口节点**（张三在哪）；**图谱 = 关系推理**（张三和谁有关系）。两者协作，缺一不可。

### 5.3 为什么不能只靠向量

```mermaid
graph TD
    Q["查询: '张三的下属是谁?'"]

    Q --> VEC_ONLY["只用向量<br/>━━━━━━━━━<br/>chunks_vdb 找相似 chunk<br/>→ 可能找到提到张三的段落<br/>→ 但不一定提到下属"]

    Q --> KG["用图谱<br/>━━━━━━━━━<br/>① 向量找张三节点<br/>② 图遍历张三的边<br/>③ 找到 张三→李四(上下级)<br/>④ 答案: 李四"]

    VEC_ONLY --> VEC_RESULT["❌ 可能漏召回<br/>（多跳推理弱）"]
    KG --> KG_RESULT["✅ 精准找到关系链"]

    style VEC_RESULT fill:#ffcdd2,stroke:#c62828
    style KG_RESULT fill:#c8e6c9,stroke:#2e7d32
```

## 六、增删改：怎么维护图谱

### 6.1 增（新文档入库）

```mermaid
graph LR
    ADD["ainsert(新文档)"] --> PARSE["解析→分块"] --> EXTRACT["抽取实体/关系"] --> MERGE["同名合并进现有图"] --> DONE["图更新"]

    style MERGE fill:#fff3e0,stroke:#e65100
```

详见第四节——自动和旧文档建立关系。

### 6.2 删（删文档，级联清理）

```mermaid
graph TD
    DEL["adelete_by_doc_id('doc-1')"]

    DEL --> FIND["找到 doc-1 的所有 chunk"]

    FIND --> TRACE["追踪受影响实体/关系<br/>━━━━━━━━━━━━━<br/>哪些节点/边的 source_id<br/>包含 doc-1 的 chunk"]

    TRACE --> CLASSIFY{"分类处理"}

    CLASSIFY -->|"实体只来自 doc-1"| DELETE["🗑️ 删除节点<br/>━━━━━━━━━━━━━<br/>graph.remove_node<br/>entities_vdb.delete<br/>级联删相关边"]

    CLASSIFY -->|"实体来自多文档"| REBUILD["🔄 用剩余 chunk 重建<br/>━━━━━━━━━━━━━<br/>rebuild_knowledge_from_chunks<br/>去掉 doc-1 来源<br/>LLM 重新生成描述<br/>重新 embed"]

    style DELETE fill:#ffcdd2,stroke:#c62828
    style REBUILD fill:#c8e6c9,stroke:#2e7d32
```

> **聪明的处理**：多来源实体不会因删一个文档而消失，而是**用剩余来源重新生成**。
