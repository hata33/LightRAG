# 多路召回全景

**项目**：LightRAG · **版本**：1.5.5 · **日期**：2026-07-10 · **作者**：15531

> 本文档完整梳理 LightRAG 查询时的**多路召回机制**：不同 mode 走哪几条路径、每条路径做什么、结果怎么合并去重、最后怎么组装上下文。全部基于源码核实（`operate.py:4315 _perform_kg_search`、`:4456 round-robin merge`、`:5024 _build_query_context`）。

---

## 一、核心概念：检索不是一路，是多路

LightRAG 的查询不是「查一次就完」，而是根据 `QueryParam.mode` **组合多条召回路径**，各路径并行执行，结果合并后送 LLM：

```mermaid
graph TB
    Q["查询 + 双层关键词<br/>ll_keywords 低层（具体实体）<br/>hl_keywords 高层（抽象主题）"]

    Q --> MODE{"mode 决定走几条路"}

    MODE -->|"local"| A["路径 A<br/>实体级图召回"]
    MODE -->|"global"| B["路径 B<br/>关系级图召回"]
    MODE -->|"hybrid"| AB["路径 A + B<br/>双层图召回"]
    MODE -->|"mix"| ABC["路径 A + B + C<br/>图召回 + 向量召回"]
    MODE -->|"naive"| C["路径 C<br/>纯向量召回"]

    A & B & AB & ABC & C --> MERGE["合并去重 + token截断 + rerank"]
    MERGE --> LLM["组装上下文 → LLM 生成"]

    style Q fill:#fffde7,stroke:#f57f17
    style MODE fill:#f3e5f5,stroke:#6a1b9a
    style MERGE fill:#e8f5e9,stroke:#2e7d32
```

## 二、四条召回路径详解

### 路径 A：实体级图召回（local 用）— `operate.py:4400`

```mermaid
graph TD
    A_IN["输入: ll_keywords<br/>例: ['张三']"]

    A_IN --> A1["① 向量库召回实体<br/>━━━━━━━━━━━━━<br/>entities_vdb.query(向量化, top_k=60)<br/>━━━ operate.py:5155 ━━━<br/>→ 返回最相似的60个实体名<br/>例: 张三/李四/A公司/CEO"]

    A1 --> A2["② 图遍历取节点详情<br/>━━━━━━━━━━━━━<br/>get_nodes_batch(entity_names)<br/>━━━ operate.py:5167 ━━━<br/>→ 每个实体的 description/source_id"]

    A2 --> A3["③ 图遍历取关联边<br/>━━━━━━━━━━━━━<br/>_find_most_related_edges<br/>get_nodes_edges_batch<br/>━━━ operate.py:5210 ━━━<br/>→ 遍历这60个实体的所有边<br/>→ 张三→李四(上下级)<br/>→ 张三→A公司(CEO)"]

    A3 --> A_OUT["输出: local_entities + local_relations<br/>实体描述按 cosine 相似度排序<br/>关系按 rank+weight 排序"]

    style A1 fill:#e3f2fd,stroke:#1565c0
    style A3 fill:#e8f5e9,stroke:#2e7d32
```

> **入口是向量库，不是图库。** 先用 ll_keywords 的 embedding 在 entities_vdb 里召回实体名，再用这些名字去图里遍历边。

### 路径 B：关系级图召回（global 用）— `operate.py:4409`

```mermaid
graph TD
    B_IN["输入: hl_keywords<br/>例: ['职责','管理']"]

    B_IN --> B1["① 向量库召回关系<br/>━━━━━━━━━━━━━<br/>relationships_vdb.query(向量化, top_k=60)<br/>━━━ operate.py:5430 ━━━<br/>→ 返回最相似的60条关系<br/>注意: 查的是关系向量库<br/>不是实体向量库"]

    B1 --> B2["② 图取关系详情<br/>━━━━━━━━━━━━━<br/>get_edges_batch<br/>→ 每条关系的 description/weight"]

    B2 --> B3["③ 取关系端点实体<br/>━━━━━━━━━━━━━<br/>从关系 src_id/tgt_id<br/>反查涉及的实体"]

    B3 --> B_OUT["输出: global_relations + global_entities<br/>关系按 cosine 相似度排序"]

    style B1 fill:#fff3e0,stroke:#e65100
```

> **路径 B 和 A 的区别**：A 先找实体再找边；B 先找关系（边），再找关系两端的实体。一个从「具体的人/物」入手，一个从「抽象的主题/关系」入手。

### 路径 C：向量 chunk 召回（只有 mix 有）— `operate.py:4437`

```mermaid
graph TD
    C_IN["输入: 原始 query<br/>例: '张三负责什么?'"]

    C_IN --> C1["① 直接向量召回文本块<br/>━━━━━━━━━━━━━<br/>chunks_vdb.query(向量化, top_k=chunk_top_k)<br/>━━━ operate.py:4284 _get_vector_context ━━━<br/>→ 不走图、不走实体<br/>→ 就是传统 RAG 的做法"]

    C1 --> C_OUT["输出: vector_chunks<br/>最相似的文本块列表<br/>标记 source='C'（vector来源）"]

    style C1 fill:#e8f5e9,stroke:#2e7d32
```

> **路径 C 是传统 RAG 的召回方式**——直接用查询语义匹配文本块。mix 模式把它和图召回叠加，作为「图谱没覆盖到时的兜底」。

### 路径 D：naive 纯向量（naive 专用）— `operate.py:5740`

```mermaid
graph TD
    D_IN["输入: 原始 query"]
    D_IN --> D1["chunks_vdb.query(直接召回 chunk)<br/>不走图、不走实体、不走关键词抽取"]
    D1 --> D_OUT["输出: chunk 列表 → 直接组装上下文"]
    style D1 fill:#fce4ec,stroke:#c62828
```

> naive 模式完全不用知识图谱，等价于传统向量 RAG。

| mode | 路径 A 实体图 | 路径 B 关系图 | 路径 C 向量chunk | 总路数 | 源码分支 |
|---|:---:|:---:|:---:|---|---|

## 四、多路结果合并：round-robin 去重

多路并行召回后，实体和关系需要合并。LightRAG 用**轮询交替（round-robin）**策略，保证两路结果均匀分布（`operate.py:4456`）：

```mermaid
graph TD
    LOCAL_E["local_entities<br/>按 cosine 排序<br/>[张三, 李四, 王五]"]
    GLOBAL_E["global_entities<br/>按 cosine 排序<br/>[A公司, 张三, B部门]"]

    LOCAL_E --> RR["round-robin 轮询交替<br/>━━━━━━━━━━━━━<br/>轮次0: local[0]=张三 ✅ → 加入<br/>轮次0: global[0]=A公司 ✅ → 加入<br/>轮次1: local[1]=李四 ✅ → 加入<br/>轮次1: global[1]=张三 ❌ 重复跳过<br/>轮次2: local[2]=王五 ✅ → 加入<br/>轮次2: global[2]=B部门 ✅ → 加入"]

    GLOBAL_E --> RR

    RR --> FINAL["final_entities<br/>[张三, A公司, 李四, 王五, B部门]<br/>━━━ 已去重 ━━━"]

    style RR fill:#fff3e0,stroke:#e65100
    style FINAL fill:#c8e6c9,stroke:#2e7d32
```

### 去重逻辑

| 类型 | 去重键 | 策略 |
|---|---|---|
| **实体** | `entity_name` | 同名只保留一次 |
| **关系** | `tuple(sorted([src, tgt]))` | 同一对端点只保留一次 |

> **为什么用 round-robin 而非简单拼接**：避免某一路完全压过另一路。local 和 global 交替取，保证两种视角的结果都有机会进入上下文。

## 五、四阶段处理流水线

召回结果经过四个阶段处理，最终变成 LLM 上下文（`operate.py:5024 _build_query_context`）：

```mermaid
graph TD
    STAGE1["阶段 1: 多路搜索<br/>━━━━━━━━━━━━━<br/>_perform_kg_search<br/>━━━ operate.py:5047 ━━━<br/>按 mode 跑路径 A/B/C<br/>返回: entities + relations + chunks"]

    STAGE1 --> STAGE2["阶段 2: token 截断<br/>━━━━━━━━━━━━━<br/>_apply_token_truncation<br/>━━━ operate.py:5067 ━━━<br/>max_entity_tokens=6000<br/>max_relation_tokens=8000<br/>超预算的按相似度截断"]

    STAGE2 --> STAGE3["阶段 3: 合并 chunk<br/>━━━━━━━━━━━━━<br/>_merge_all_chunks<br/>━━━ operate.py:5074 ━━━<br/>三类 chunk 来源合并:<br/>① 实体 source_id 反查的 chunk<br/>② 关系 source_id 反查的 chunk<br/>③ 路径 C 向量召回的 chunk"]

    STAGE3 --> STAGE4["阶段 4: 组装上下文<br/>━━━━━━━━━━━━━<br/>_build_context_str<br/>━━━ operate.py:5096 ━━━<br/>entities_context + relations_context<br/>+ merged_chunks → 最终文本<br/>max_total_tokens=30000 截断<br/>可选 enable_rerank 重排"]

    STAGE4 --> LLM["→ LLM 生成答案"]

    style STAGE1 fill:#e3f2fd,stroke:#1565c0
    style STAGE2 fill:#fff3e0,stroke:#e65100
    style STAGE3 fill:#e8f5e9,stroke:#2e7d32
    style STAGE4 fill:#f3e5f5,stroke:#6a1b9a
```

### chunk 的三个来源合并（阶段 3）

```mermaid
graph LR
    subgraph CHUNK_SOURCES["chunk 来源"]
        direction TB
        CS1["来源 ①: 实体反查<br/>━━━━━━━━━━━━━<br/>filtered_entities 的 source_id<br/>→ text_chunks.get_by_ids<br/>标记 source='E'"]
        CS2["来源 ②: 关系反查<br/>━━━━━━━━━━━━━<br/>filtered_relations 的 source_id<br/>→ text_chunks.get_by_ids<br/>标记 source='R'"]
        CS3["来源 ③: 向量召回<br/>━━━━━━━━━━━━━<br/>路径 C 的 vector_chunks<br/>标记 source='C'"]
    end

    CS1 & CS2 & CS3 --> MERGE_C["合并去重<br/>━━━━━━━━━━━━━<br/>同一个 chunk 可能<br/>被多路命中<br/>frequency 记录命中次数"]

    MERGE_C --> RANK_C["按来源+频次排序<br/>━━━━━━━━━━━━━<br/>多路命中的 chunk 排前"]

    style MERGE_C fill:#c8e6c9,stroke:#2e7d32
```

> **chunk_tracking** 记录每个 chunk 被哪条路径命中（source 标记）和命中次数（frequency）。多路命中的 chunk 排名更靠前——因为它既和实体相关、又和关系相关、还和查询语义相关。

| 组件 | 角色 | 用在哪条路径 |
|---|---|---|

## 九、mode 选择建议

```mermaid
graph TD
    Q["查询场景"]

    Q --> Q1{"需要多跳关系推理?"}
    Q1 -->|"是"| Q2{"需要全局综述?"}
    Q1 -->|"否，只要语义匹配"| NAIVE["naive<br/>纯向量，最快"]

    Q2 -->|"都要"| MIX["mix（默认推荐）<br/>三路全开 + rerank"]
    Q2 -->|"只要具体关系"| LOCAL["local<br/>实体级图召回"]
    Q2 -->|"只要全局主题"| GLOBAL["global<br/>关系级图召回"]
    Q2 -->|"要细节+综述<br/>不需要向量兜底"| HYBRID["hybrid<br/>双层图召回"]

    style MIX fill:#c8e6c9,stroke:#2e7d32
    style NAIVE fill:#fce4ec,stroke:#c62828
```

### 性能与效果权衡

| mode | 召回路数 | LLM 调用 | 效果 | 延迟 |
|---|---|---|---|---|
| **naive** | 1（向量） | 1（关键词可选跳过） | 基础 | ⚡ 最快 |
| **local** | 1（图） | 2（关键词+生成） | 实体级 | 快 |
| **global** | 1（图） | 2 | 主题级 | 快 |
| **hybrid** | 2（图×2） | 2 | 均衡 | 中 |
| **mix** | 3（图×2+向量） | 2 | ⭐ 最佳 | 中 |

> **mix 是默认模式**——虽然三路召回，但关键词抽取只调 1 次 LLM，三路搜索是并行的，所以延迟并不比 hybrid 高多少。
