# LightRAG 解析流水线全流程详解

**项目**：LightRAG · **版本**：1.5.5 · **日期**：2026-07-08 · **作者**：15531

> 本文档是**当前代码**(`pipeline.py` / `operate.py` / `parser/` / `chunker/`)里解析流水线的**完整流程拆解**,从文档进入系统到知识入库,覆盖每一个阶段、状态机转换、并发模型、降级策略与失败处理。所有函数名、行号、参数均来自源码。

---

## 一、全景:三条级联队列

LightRAG 的文档处理是一个**三级流水线**(源码:`pipeline.py:1143-1181` `_run_pipeline_batch`):

```
                     ┌─────────────────────────────────────────────────────────┐
                     │              apipeline_process_enqueue_documents          │
                     │     (入口循环:扫描 PENDING → 调度 _run_pipeline_batch)    │
                     └────────────────────────────┬────────────────────────────┘
                                                   │ 按 batch 取 PENDING 文档
                                                   ▼
         ┌─────────────────────────────────────────────────────────────────────┐
         │                       _run_pipeline_batch                            │
         │        三条级联队列 + 三组 worker,用 asyncio.Queue 串联                │
         └─────────────────────────────────────────────────────────────────────┘

   Layer 1: 解析(parse)         Layer 2: 多模态分析(analyze)      Layer 3: 抽取+入库(process)
   ┌──────────────────┐          ┌──────────────────┐             ┌────────────────────┐
   │  _parse_worker   │──q_analyze──▶ _analyze_worker │──q_process──▶ _process_worker    │
   │                  │          │                  │             │                    │
   │ native/mineru/   │          │ analyze_multimodal│            │ process_single_doc │
   │ docling/legacy   │          │ (VLM 分析图片)    │             │ chunk→extract→merge│
   └──────────────────┘          └──────────────────┘             └────────────────────┘
        ▲ 每引擎组独立队列              ▲ 单队列                            ▲ 受 semaphore 限流
   max_parallel_parse_*           queue_size_analyze              max_parallel_insert
```

**关键设计**:
- **解析层按引擎分组**:native / mineru / docling 各有独立队列和并发上限(因为外部服务调用成本/速率不同)。
- **分析层是可选的**:只有文档含图片/绘图且开启 VLM 时才真正工作,否则直通。
- **处理层全局信号量**:`max_parallel_insert`(默认 3,建议上限 10)控制 KG 抽取的 LLM 并发,防击穿下游 LLM。

## 二、阶段零:文档进入系统(Enqueue)

### 入口
- **SDK**:`rag.ainsert(text)` —— 固定走 F 分块,内部调 `apipeline_enqueue_documents`。
- **服务端**:`POST /documents/text` / `/documents/upload` / `/documents/scan` —— 调 `apipeline_enqueue_documents` + `process_options`(支持 F/R/V/P)。

### `apipeline_enqueue_documents`(`pipeline.py:232`)做的事

1. **写入 `full_docs`**(原文/解析产物)—— **先于** `doc_status` 写入(写顺序契约,保证处理循环一定能读到内容)。
2. **写入 `doc_status`**:状态 = `PENDING`,记录 `file_path` / `content_hash` / `process_options` 等。
3. **触发处理循环**:若当前不忙,直接启动;若忙,设置 `request_pending` 标志让循环下一轮感知。

### 并发契约(`pipeline_status` 共享字典)
| 字段 | 含义 |
|---|---|
| `busy` | 任何处理中状态(含破坏性任务) |
| `destructive_busy` | clear/delete 执行中(独占,拒绝并发上传) |
| `scanning` / `scanning_exclusive` | scan 任务运行中 / 分类阶段(独占) |
| `pending_enqueues` | 已预留但 bg 任务未完成的上传数 |
| `request_pending` | 催促循环重新查询 doc_status 的标志 |

> 允许「上传中 + 处理中」并发:新文档落 PENDING → 循环处理完当前 batch 查 `request_pending` → 拿到新行。

## 三、阶段一:解析(Parse) —— Layer 1

### 入口循环:`apipeline_process_enqueue_documents`(`pipeline.py:927`)

```
入口检查:
  ├─ 已 busy? → 设 request_pending,返回(让现有循环接管)
  └─ 不忙? → busy=True → 进入 _run_pipeline_batch
     循环:
       ├─ 查 doc_status 取 PENDING 文档(batch)
       ├─ 无文档? → busy=False,退出
       └─ 有文档? → _run_pipeline_batch(batch)
                       └─ 跑完后检查 request_pending → 有则重新查询,无则退出
```

### `_parse_worker`(`pipeline.py:1546`)

**职责**:把一个原始文档(文件/文本)解析成 `ParseResult`(纯文本/Markdown + 结构化 blocks)。

```
_parse_worker(doc_id, status_doc):
  1. 读取 full_docs 取 content_data
  2. 解析 process_options 确定引擎(native/mineru/docling/legacy)
  3. 状态置 PARSING
  4. 调用 BaseParser.parse(ctx) → ParseResult
       ├─ native:  本地解析(docx/markdown/textpack)
       ├─ legacy:  本地基础解析(pdf/docx/pptx/xlsx/...)
       ├─ mineru:  HTTP 调 MinerU 服务(版面+OCR+表格+公式)
       └─ docling: HTTP 调 Docling 服务
  5. _persist_parsed_full_docs: 持久化解析结果到 full_docs
  6. 去重检查:若 content_hash 与已有文档相同 → _mark_duplicate_after_parse(短路)
  7. 投入 q_analyze(进入 Layer 2)
  8. 异常 → 状态置 FAILED + 错误信息
```

**ParseResult 核心字段**(`base.py:108`):
| 字段 | 内容 |
|---|---|
| `content` | 解析出的纯文本/Markdown |
| `blocks_path` | 结构化块文件路径(P 分块依赖此) |
| `parse_format` | 格式标记 |
| `parse_engine` | 实际使用的引擎 |

> 这就解释了你之前遇到的 `blocks_path is empty` 警告:某些引擎(如 native 对纯文本 docx)不产 blocks,导致 P 分块降级为 R。

### 外部引擎的统一模板(`external/_base.py`)

MinerU/Docling 共享 `ExternalParserBase`,固定流程:
```
resolve(源路径/输出目录)
  → is_bundle_valid(缓存命中?跳过下载)
  → 否则 download_into(调外部服务拿原始包)
  → build_ir(原始包 → IRDoc 标准中间表示)
  → write_sidecar(写 sidecar + blocks.jsonl)
  → persist full_docs + archive 源文件
```

## 四、阶段二:多模态分析(Analyze) —— Layer 2

### `_analyze_worker`(`pipeline.py:1869`)

**职责**:对解析结果中的图片/绘图做 VLM 分析,提取文字描述,增强图谱的多模态实体。

```
_analyze_worker(ctx):
  从 q_analyze 取文档
  → analyze_multimodal(parsed_data)
       ├─ 识别图片/图表/绘图(item.type == image/drawing)
       ├─ 调用 VLM 角色 LLM 生成描述(带 JSON 格式校验重试)
       ├─ 文本类:调 EXTRACT 角色补充抽取
       └─ 结果回写增强 content
  → 投入 q_process(进入 Layer 3)
```

**何时真正工作**:仅当文档含图片且配置了 VLM(`llm_roles.py` 的 `VLM` 角色)时。纯文本文档此层近乎直通。

**关键函数**:`analyze_multimodal`(`pipeline.py:3247`),内含 `_analyze_drawing`(绘图分析)、`_analyze_text_modality`(文本模态补抽)、`_run_json_conformance_retry`(格式重试)。

## 五、阶段三:抽取与入库(Process) —— Layer 3

### `_process_worker`(`pipeline.py:1998`)→ `process_single_document`(`pipeline.py:2035`)

这是**核心状态机**,驱动 `PROCESSING → PROCESSED`,含两道失败边界(extract / merge)。

#### 5.1 分块(Chunking)

```
读取 process_options.chunking 策略:
  ├─ "F" → chunking_by_fixed_token        (固定 token 大小)
  ├─ "R" → chunking_by_recursive_character (递归字符,带分隔符层级)
  ├─ "V" → chunking_by_semantic_vector     (语义向量,embedding 聚类)
  └─ "P" → chunking_by_paragraph_semantic  (段落语义,依赖 blocks_path)
           └─ blocks_path 为空? → WARNING + 降级为 R(递归字符)
```

分块后写入 `chunks` KV 存储 + `chunks_vdb`(向量库)。`chunk_results` 待传给抽取。

**SDK `ainsert` 的限制**:只走 F 分块(不传 `process_options`)。要 R/V/P 必须走服务端 `apipeline_enqueue_documents` + `process_options`。

#### 5.2 实体/关系抽取(Extract)

```
_process_extract_entities(chunks, ...):   ← 并发执行(信号量限流)
  ├─ 遍历 chunks,对每个 chunk 调 extract_entities  ← operate.py:3320
  │     ├─ 构造抽取 prompt(PROMPTS["entity_extraction"])
  │     ├─ 调 EXTRACT 角色 LLM(带缓存查 LLM_RESPONSE_CACHE)
  │     ├─ 解析 LLM 返回的 JSON(实体+关系)
  │     └─ gleaning 循环(最多 MAX_GLEANING 轮补抽)
  └─ 汇总 chunk_results(每 chunk 的实体/关系列表)
```

**并发模型**:`_process_with_semaphore`(`operate.py:3718`)用信号量控制每个 chunk 的抽取并发,避免 LLM 过载。

**skip_kg 选项**:`process_options` 可设 `skip_kg=True`,跳过抽取只入库 chunks(适合纯向量检索场景)。

#### 5.3 合并入库(Merge)

```
merge_nodes_and_edges(chunk_results, ...):   ← operate.py:2914
  ├─ _locked_process_entity_name:        实体级 keyed lock(去重合并)
  │     ├─ 同名实体描述合并 + LLM 重摘要(_summarize_descriptions)
  │     ├─ 合并 source_id(来自哪些 chunk/doc)
  │     └─ entities_vdb.upsert(embedding)   向量入库
  ├─ _locked_process_edges:              关系级 keyed lock
  │     ├─ 同源-目标关系合并 + LLM 重摘要
  │     ├─ 合并 weight / source_id
  │     └─ relationships_vdb.upsert(embedding) 向量入库
  └─ knowledge_graph_inst.upsert_node/upsert_edge  图谱入库
```

**并发控制**:`keyed lock` —— 每个实体名/关系键一把锁,允许多个不同实体并行合并,同名实体串行(防竞争)。这是 LightRAG 处理大规模语料不锁死的关键。

#### 5.4 收尾

```
_insert_done:   ← lightrag.py
  ├─ 状态置 PROCESSED
  ├─ 记录 processing_time / chunk_count
  ├─ 刷盘索引(index_done_callback)
  └─ 更新 pipeline_status(本批进度)
```

## 六、状态机全景(`DocProcessingStatus`)

```
               ┌──────────┐
   enqueue ──▶ │ PENDING  │
               └────┬─────┘
                    │ parse_worker 取走
                    ▼
               ┌──────────┐  解析异常
               │ PARSING  │──────────▶ FAILED ◀── (任何阶段失败)
               └────┬─────┘                       │
                    │ analyze_worker               │ scan 重置
                    ▼                              ▼
               ┌──────────┐  分析异常         ┌──────────┐
               │ANALYZING │──────────▶ FAILED │ PENDING  │(重试)
               └────┬─────┘                   └──────────┘
                    │ process_worker
                    ▼
               ┌──────────┐  抽取/合并异常
               │PROCESSING│──────────▶ FAILED
               └────┬─────┘
                    │ merge 完成 + _insert_done
                    ▼
               ┌──────────┐
               │PROCESSED │
               └──────────┘
```

**scan 的容错**:`POST /documents/scan` 会把卡在 `PARSING/ANALYZING/PROCESSING/FAILED` 的文档重置为 `PENDING` 重新处理(你日志里看到的 `Reset 1 documents from PARSING/ANALYZING/PROCESSING/FAILED to PENDING status` 就是这个)。

## 七、检索流水线(Query) —— 读取侧

入库完成后,查询走 `operate.py` 的另一套流程:

```
aquery(query, QueryParam(mode))          lightrag.py:2077
  └─▶ extract_keywords_only / get_keywords_from_query    operate.py:4001/4156
        │   调 KEYWORDS 角色 LLM 抽取关键词
        │   → {low_level_keywords, high_level_keywords}
        ▼
      [mode 分支]
        ├─ local  → kg_query(low_level)     operate.py:3786
        ├─ global → kg_query(high_level)
        ├─ hybrid → kg_query(low + high)
        ├─ mix   → kg_query + naive_query 叠加
        ├─ naive  → naive_query()           operate.py:5728(纯向量,不走图)
        └─ bypass → 跳过检索
              │
              ▼
        kg_query 内部(_perform_kg_search:4315):
          ├─ entities_vdb.query()        向量召回实体
          ├─ relationships_vdb.query()   向量召回关系
          ├─ graph.get_node_edges()      图遍历取子图
          ├─ _find_most_related_*        找最相关边/实体
          ├─ chunks_vdb.query()          召回相关 chunk
          ├─ _merge_all_chunks()         合并去重
          └─ (可选) rerank               重排序
              │
              ▼
        _build_context_str()             组装上下文(实体/关系/chunk)
              │
              ▼
        QUERY 角色 LLM.complete(context + query)  → 答案(可流式 SSE)
```

## 八、关键设计要点速查

| 设计 | 作用 | 代码位置 |
|---|---|---|
| **三级级联队列** | 解析/分析/处理解耦,各层独立限速 | `_run_pipeline_batch:1143` |
| **按引擎分组队列** | 外部引擎(mineru/docling)不拖慢本地引擎 | `parse_queues:1167` |
| **keyed lock 合并** | 同名实体串行、不同实体并行,防竞争不锁死 | `merge_nodes_and_edges:2914` |
| **状态机 + 失败边界** | 每阶段独立失败标记,scan 可重试 | `process_single_document:2035` |
| **full_docs 先于 doc_status 写** | 保证处理循环一定能读到内容 | `apipeline_enqueue_documents:232` |
| **request_pending 催促** | 允许上传与处理并发 | `apipeline_process_enqueue_documents:927` |
| **P 分块降级 R** | blocks 缺失时不丢内容,自动退化 | `paragraph_semantic.py:2093` |
| **LLM 响应缓存** | 抽取/查询结果复用,省成本 | `LLM_RESPONSE_CACHE` KV 存储 |
| **gleaning 多轮补抽** | LLM 首轮漏抽的实体,多轮补全 | `extract_entities:3320` |

## 九、一图总结完整生命周期

```
【文档】 ──enqueue──▶ PENDING
   │
   ▼  parse_worker (Layer 1)
  PARSING ──解析引擎──▶ [纯文本/MD + blocks.jsonl]
   │        native / legacy / mineru / docling
   ▼  analyze_worker (Layer 2)
  ANALYZING ──VLM──▶ [多模态增强]
   │        (可选,含图片时)
   ▼  process_worker (Layer 3)
  PROCESSING
   ├─ 分块: F/R/V/P ──▶ chunks_vdb
   ├─ 抽取: extract_entities ──▶ entities/relationships
   └─ 合并: merge_nodes_and_edges ──▶ 图谱 + 实体/关系向量库
   │
   ▼
  PROCESSED
   │
   ▼  aquery (读取侧)
  关键词抽取 → mode 检索 → rerank → LLM 生成 → 答案
```
