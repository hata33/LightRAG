# LightRAG 技术栈与能力全景

**项目**：LightRAG · **版本**：1.5.5 · **日期**：2026-07-08 · **作者**：15531

> 本文档回答两个问题：**「这个项目用了哪些东西」** 与 **「这个项目有哪些能力」**。是一份覆盖后端、前端、存储、模型、部署的技术清单与能力速查，配合 `能力与使用指南.md`（偏使用）与 `核心执行链路与架构速览文档.md`（偏源码）阅读。

---

## 一、一句话定位

LightRAG 是 HKUDS 开源的**知识图谱驱动的检索增强生成（RAG）框架**：从文档中抽取实体与关系构建知识图谱，再以 local / global / hybrid / mix / naive 五种模式做双层检索问答，存储、模型、解析器全可插拔，既能零依赖单机运行，也能拼成生产级集群。

---

## 二、整体架构

```
用户 ──→ WebUI (React 19) ──→ FastAPI 服务 (Python) ──→ LightRAG 引擎
                                                          │
                        ┌─────────────────────────────────┼─────────────────────┐
                        ▼                                 ▼                     ▼
                   文档处理管线                      知识图谱 + 向量库      多模式检索
                 (解析→分块→抽取→入库)              (实体/关系/chunk)       (local/global/hybrid/mix/naive)
```

## 四、可插拔存储后端（4 类 / 22+ 实现）

LightRAG 最核心的设计——**4 种存储类型，每种都能自由换后端**。默认零依赖运行（全 JSON 文件 + NanoVectorDB + NetworkX）。

| 存储类型 | 作用 | 可选后端 |
|---|---|---|
| **KV_STORAGE** | LLM 响应缓存、文本块、文档信息 | JSON · Redis · PostgreSQL · MongoDB · OpenSearch |
| **VECTOR_STORAGE** | 实体/关系/chunk 的向量嵌入 | NanoVectorDB · Milvus · PostgreSQL(pgvector) · FAISS · Qdrant · MongoDB · OpenSearch |
| **GRAPH_STORAGE** | 实体-关系图结构 | NetworkX · Neo4j · PostgreSQL(Apache AGE) · MongoDB · Memgraph · OpenSearch |
| **DOC_STATUS_STORAGE** | 文档处理状态跟踪 | JSON · Redis · PostgreSQL · MongoDB · OpenSearch |

> 一体化后端（PostgreSQL / MongoDB / OpenSearch）可单独承载全部四类存储，运维最简；Neo4j + Milvus 等专用组合适合大规模场景。

| 角色 | 用途 |
|---|---|
| `EXTRACT` | 实体/关系抽取（建议用强模型保质量） |

| `KEYWORDS` | 查询关键词抽取（可用快模型省成本） |
| `VLM` | 视觉语言模型，处理多模态图像 |

## 六、核心能力

### 6.1 文档处理管线（`pipeline.py`）

- **多格式解析**：PDF / DOCX / PPTX / XLSX / Markdown / HTML
- **多解析引擎**：`legacy`（内置）/ `native`（增强）/ `MinerU`（外部 HTTP，学术论文友好）/ `Docling`（外部 HTTP）
- **四种分块策略**：token-size (F) / recursive-character (R) / semantic-vector (V) / paragraph-semantic (P)
- **多模态**：支持图像分析（需 VLM 角色），把图片中的实体也抽进图谱
- **并发管线**：细粒度锁机制（`pipeline_status` + `get_namespace_lock`），支持「上传中同时处理」「扫描与上传并存」等并发场景

### 6.2 知识图谱构建（`operate.py`）

从文档抽取**实体 + 关系 + 声明（claims）**，构建带类型、带权重的图，支持去重合并（`merge_nodes_and_edges`）。

### 6.3 五种检索模式

| 模式 | 说明 | 适用场景 |
|---|---|---|
| **local** | 聚焦具体实体的上下文检索 | 针对性细节问答 |
| **global** | 基于社区/摘要的广度检索 | 全局综述、主题汇总 |
| **hybrid** | local + global 融合 | 兼顾细节与综述 |
| **naive** | 纯向量检索，不走图 | 传统 RAG 行为 |
| **mix** | KG + 向量深度融合（**配 reranker 最佳**，默认模式） | 复杂多跳问题 |

### 6.4 其他能力

- **重排序（Rerank）**：Cohere / Jina / Ali / 通用 HTTP reranker
- **引用溯源**：答案返回来源 chunk / 文档
- **图谱编辑**：实体/关系 CRUD、合并、按文档删除
- **Workspace 数据隔离**：同一实例按 workspace 分区
- **自定义 KG 导入**：`ainsert_custom_kg` 直接导入已有三元组
- **多语言**：i18next 前端国际化
- **存储迁移**：`storage_migrations.py` 支持历史数据自动迁移
