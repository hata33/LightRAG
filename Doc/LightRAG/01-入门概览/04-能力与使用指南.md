# LightRAG 能力与使用指南

**项目**：LightRAG · **版本**：1.5.5（文档版本 1.26.705.8600）· **日期**：2026-07-05 · **作者**：15531

> 本文档**优先讲能力与效果**：LightRAG 能做什么、每项能力达成什么效果、以及具体怎么用。

---

## 一、它能做什么（核心能力）

LightRAG 是一个**知识图谱驱动的 RAG 框架**，围绕"建库—问答—管理"提供端到端能力：

| 能力域 | 能力点 | 实现效果 |
|--------|--------|----------|
| **文档摄入** | 多格式上传/批量文本/目录扫描 | 把 PDF、Word、PPT、Excel、Markdown、图片等异构资料统一纳管 |
| **多模态理解** | MinerU/Docling/RAG-Anything 集成 | 正确解析表格、公式、图片等非文本内容 |
| **智能分块** | F/R/V/P 四种分块策略 | 按需选择 token/递归/语义/段落切分，兼顾召回精度与结构保留 |
| **知识图谱构建** | LLM 自动抽取实体与关系 | 把散落文档组织成可推理的实体-关系网络 |
| **双层检索问答** | local/global/hybrid/mix/naive/bypass 六模式 | 细节事实、全局综述、多跳推理都能精准作答 |
| **结果重排** | Reranker（Cohere/Jina/Ali/通用） | 显著提升混合查询的答案相关性 |
| **引用溯源** | 返回来源 chunk/文档 | 答案可追溯，增强可信度 |
| **图谱编辑** | 实体/关系 CRUD、合并、按文档删除 | 知识库可维护、可纠错 |
| **存储可插拔** | 20+ 后端（KV/向量/图/状态） | 从单机 JSON 文件平滑演进到分布式集群 |
| **模型可插拔** | 13+ LLM、9+ Embedding 提供商 | 不绑定任何一家模型，支持全本地化 |
| **角色化模型** | EXTRACT/QUERY/KEYWORDS/VLM 四角色 | 抽取、问答、关键词、视觉各用最合适的模型 |
| **API 服务** | FastAPI REST + Ollama 兼容 | 一行命令起服务，前端/第三方可直接对接 |
| **可视化** | React Web UI + Sigma.js 图谱 | 非技术用户也能上传、问答、浏览知识图谱 |
| **评估与追踪** | RAGAS 评估 + Langfuse 追踪 | 量化检索质量、监控 LLM 调用链 |
| **运维工具** | 缓存清理、向量重建、迁移、哈希 | 日常运维与数据治理 |

## 二、典型使用场景与效果

### 场景 1：企业私有知识库问答
**痛点**：内部文档多、传统搜索找不到、向量 RAG 多跳推理弱。
**效果**：上传文档 → 自动建图 → hybrid/mix 模式问答，支持跨文档推理与全局综述，答案带引用。

### 场景 2：科研/法律/医疗文献分析
**痛点**：大量长文档、术语关系复杂。
**效果**：语义/段落分块保留结构，KG 抽取理清实体关系，global 模式做主题综述。

### 场景 3：多模态资料处理
**痛点**：PDF 含表格/公式/图片。
**效果**：多模态解析 + VLM 角色理解图像，表格与公式不丢失。

### 场景 4：合规/数据不出域
**痛点**：不能用云端 LLM。
**效果**：Ollama/vLLM 本地模型 + JSON/Postgres 本地存储，全链路私有化。

### 场景 5：生产级扩展
**痛点**：单机扛不住大规模语料。
**效果**：切换 Postgres/Mongo/OpenSearch 一体后端或 Neo4j+Milvus 专用后端，多 Worker 水平扩展。

## 三、如何使用（快速上手）

### 3.1 三种使用方式总览

| 方式 | 适合 | 上手难度 |
|------|------|----------|
| **Web UI** | 非技术用户、快速体验 | ★ |
| **REST API** | 应用集成、第三方对接 | ★★ |
| **Python SDK** | 二次开发、深度定制 | ★★★ |

### 3.2 安装（5 分钟）
```bash
# 推荐用 uv
curl -LsSf https://astral.sh/uv/install.sh | sh
uv pip install -e ".[api]"          # 含 Web API 与文档解析
# 可选：uv pip install -e ".[offline]"   # 全部后端与模型

cp env.example .env                 # 配置 LLM/嵌入/存储
```

> **`-e` 是什么意思？安装包从哪里来？**
>
> - `-e` = editable（可编辑模式），安装的**不是**复制到 `site-packages` 的副本，而是一个**指向你项目源码的链接**。你改 `lightrag/` 下的任何 `.py` 文件，改动**立即生效**，不需要重新安装。
> - 安装源是**当前项目根目录**的 `pyproject.toml`——它声明了 `lightrag` 包的内容（`[tool.setuptools.packages.find]` 里的 `include = ["lightrag*"]`）。
> - `.[api]` 的 `.` = 当前目录，`[api]` = 安装 `pyproject.toml` 里 `[project.optional-dependencies]` 的 `api` 额外依赖组（FastAPI、文档解析库等）。
> - 不带 `-e`（`uv pip install ".[api]"`）会把代码复制一份到 `site-packages`，改源码不生效。**开发场景一定要带 `-e`**。

最小 `.env`（OpenAI 为例）：
```env
LLM_BINDING=openai
LLM_BINDING_HOST=https://api.openai.com/v1
LLM_MODEL=gpt-4o-mini
LLM_API_KEY=sk-xxx
EMBEDDING_BINDING=openai
EMBEDDING_MODEL=text-embedding-3-large
EMBEDDING_DIM=3072
```

### 3.3 方式一：启动 Web UI
```bash
lightrag-server --host 0.0.0.0 --port 9621
# 浏览器访问 http://localhost:9621/webui/
```
界面操作：**文档**上传 → **知识库**选模式提问 → **图谱**可视化浏览。

### 3.4 方式二：REST API
```bash
# 1) 摄入
curl -X POST http://localhost:9621/documents/text \
  -H "Authorization: Bearer <JWT>" -H "Content-Type: application/json" \
  -d '{"text":"LightRAG 用知识图谱做双层检索。","track_id":"t1"}'

# 2) 查询（流式）
curl -X POST http://localhost:9621/query/stream \
  -H "Authorization: Bearer <JWT>" -H "Content-Type: application/json" \
  -d '{"query":"检索范式是什么？","mode":"hybrid"}'

# 3) 仅检索（不生成，用于评估/可视化）
curl -X POST http://localhost:9621/query/data \
  -H "Authorization: Bearer <JWT>" -H "Content-Type: application/json" \
  -d '{"query":"...","mode":"mix"}'
```

### 3.5 方式三：Python SDK
```python
import asyncio
from lightrag import LightRAG, QueryParam

async def main():
    rag = LightRAG(working_dir="./rag_storage")
    await rag.initialize_storages()

    await rag.ainsert("你的文档内容...")          # 摄入
    ans = await rag.aquery(                       # 问答
        "你的问题", param=QueryParam(mode="hybrid")
    )
    print(ans)

asyncio.run(main())
```

> **SDK 能改源码吗？** —— **能**。SDK 不是打包成不可改的库，就是项目源码本身。
>
> - 用 `uv pip install -e ".[api]"` 安装后，`lightrag` 包指向你的项目源码目录，改 `lightrag/` 下任何文件立即生效。
> - `from lightrag import LightRAG` 导入的就是你项目里的 `lightrag/lightrag.py`，不是别处的副本。
> - SDK 方式和 Web UI 方式跑的是**同一套代码**，区别只是入口：SDK 是 `import` 后在代码里调，Web UI 是起 FastAPI 服务通过浏览器操作。
> - **本质**：LightRAG 是一个 Python 库，API 服务和 Web UI 只是在这个库上包了一层。SDK 方式 = 直接用这个库，最大灵活度。

### 3.6 Docker 一键部署
```bash
docker compose up -d                                   # 基础
docker compose -f docker-compose-full.yml up -d        # 含 Postgres + Neo4j 全栈
```

## 四、查询模式怎么选（关键效果指南）

| 你想要的效果 | 选哪个模式 | 说明 |
|------------|-----------|------|
| 默认均衡、最常用 | `mix`（默认） | 图检索 + 向量召回，配合 Rerank |
| 针对具体实体/细节 | `local` | 低层实体检索 |
| 全局综述/主题汇总 | `global` | 高层主题检索 |
| 兼顾细节与综述 | `hybrid` | local + global |
| 传统向量 RAG 行为 | `naive` | 不用知识图谱 |
| 直接问 LLM（不检索） | `bypass` | 跳过检索 |

调参要点：`TOP_K`（召回数）、`COSINE_THRESHOLD`（相似度阈值）、`RERANK_MODEL`（重排）、`history_turns`（多轮对话）。

### 4.1 各模式的业务实际区别

| 业务场景举例 | 适合模式 | 为什么 |
|---|---|---|
| 「张三的职务是什么？」 | **local** | 问的是**具体实体的属性**，从实体节点出发找关联边即可 |
| 「公司整体的组织架构是怎样的？」 | **global** | 问的是**全局综述**，需要从关系/主题维度召回 |
| 「张三和李四之间有什么关系？」 | **local** 或 **hybrid** | 具体关系用 local，想了解更多背景用 hybrid |
| 「这个产品线的历史发展脉络？」 | **global** | 主题级综述，需要高层关系网络 |
| 不知道问题属于细节还是综述 | **mix**（默认） | 三路全开，自动覆盖 |
| 「这段话是什么意思？」（纯语义匹配） | **naive** | 不需要图谱推理，纯向量相似就够了 |
| 文档库里没有相关信息，想直接问 LLM | **bypass** | 跳过检索，等于直接用 ChatGPT |

### 4.2 各模式的实现原理区别

不同模式的本质区别在于：**走几条召回路径 + 用什么关键词层**。

```
每个查询先经 KEYWORDS 角色 LLM 抽取两类关键词:
  low_level (ll): 具体词，如「张三」「A公司」
  high_level (hl): 抽象词，如「职责」「管理架构」
```

| 模式 | 召回路径 | 具体做什么 |
|---|---|---|
| **local** | 路径 A（1路） | ll 关键词 → `entities_vdb`（实体向量库）召回实体 → 图遍历该实体的关联边 |
| **global** | 路径 B（1路） | hl 关键词 → `relationships_vdb`（关系向量库）召回关系 → 取关系端点实体 |
| **hybrid** | 路径 A+B（2路） | local + global 都跑，结果 round-robin 轮询合并 |
| **mix** | 路径 A+B+C（3路） | hybrid 基础上**额外加路径 C**：原始 query → `chunks_vdb` 直接向量召回文本块。图召回 + 向量召回双保险 |
| **naive** | 路径 D（1路） | 原始 query → `chunks_vdb` 直接召回文本块，**不走图、不走实体关系、不抽关键词** |
| **bypass** | 0 路 | 不检索，直接把问题交给 LLM |

**三个关键区别**：
1. **范围不同**：local 从「具体实体」入手，global 从「抽象关系」入手，mix 两者+向量全覆盖
2. **路径数量不同**：local/global=1路，hybrid=2路，mix=3路，路径越多覆盖越广但越慢
3. **用不用图谱不同**：local/global/hybrid/mix 都用图谱做关系推理；naive 不用图谱（纯向量）；bypass 什么都不用

> **为什么 mix 效果最好但最慢？** 它跑 3 条路径（实体图召回 + 关系图召回 + 向量 chunk 召回），覆盖最全面，但每路都要 embedding + 图遍历。速度换准确率的典型 trade-off。

> 详细的召回流程图见 `03-解析与存储/07-多路召回全景.md`。

## 五、存储与模型怎么选

| 规模/场景 | 推荐后端 | 推荐模型 |
|----------|----------|----------|
| 个人/原型 | JSON + NanoVectorDB + NetworkX（默认零配置） | OpenAI / Ollama 本地 |
| 生产统一 | PostgreSQL（+pgvector+AGE）/ MongoDB / OpenSearch | OpenAI / Claude / Gemini |

> **AGE 是什么？** —— Apache AGE（A Graph Extension），是 PostgreSQL 的**图数据库扩展**。它让你在同一个 PG 实例里同时拥有：
> - **pgvector** = 向量存储能力（实体/关系/chunk 的 embedding）
> - **Apache AGE** = 图存储能力（知识图谱的节点和边，支持 Cypher 查询）
> - 普通 SQL 表 = KV 存储（文档/缓存/状态）
>
> 这样一个 PG 实例就能承载 LightRAG 的**全部四类存储**（KV + 向量 + 图 + 状态），运维最简。
> 对应配置 `LIGHTRAG_GRAPH_STORAGE=PGGraphStorage`（用的就是 AGE）。
> 如果图用 Neo4j，则 PG 只需要 pgvector（不需要 AGE），对应 `docker-compose-demo.yml` 的方案。
| 大规模图谱 | Neo4j / Memgraph | 同上 |
| 大规模向量 | Milvus / Qdrant / FAISS | 同上 |
| 离线/合规 | 全本地后端 | Ollama / vLLM / LM Deploy |

> 一体化后端（Postgres、Mongo、OpenSearch）可单独承载全部四类存储，运维最简。
